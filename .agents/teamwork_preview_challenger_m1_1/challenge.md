# 对抗性验证与评审报告 — Agora 网格宕机 RPC 降级边界

## 1. 对抗性验证摘要

- **测试对象**：eCOS 工作流引擎之 RPC 通信降级与熔断机制
- **测试环境**：模拟 Agora 网关服务不可达、假死（丢包挂起）、高延迟等故障
- **核心结论**：**系统未能实现零延迟熔断降级。在 Agora 宕机时，系统存在严重的延迟累加与长达数十秒的进程挂起风险。**
- **安全风险等级**：**HIGH** (高危)。

---

## 2. 故障注入实验结果

为了深入验证 ecos 编排引擎与 RPC 降级机制在网络挂死时的表现，我们编写并执行了对抗性测试脚本 `tests/test_adversarial_circuit_breaker.py`，模拟了以下两个关键故障场景并运行了测试：

### 实验一：Swarm 后端在网络挂起时的延迟累加 (backends/swarm.py)
- **模拟故障**：Agora 网关假死（例如网络丢包导致握手超时），当 client 发起请求时触发 1.5 秒连接挂起，最后超时抛出 `ConnectTimeout`。工作流定义包含 3 个 Swarm 执行步骤。
- **理论预期**：若具备智能熔断或针对 Agora 宕机状态的缓存，在第 1 步超时后应立即记录故障，后续 2 步瞬时触发熔断，直接 fallback 本地 CLI，整体耗时应控制在 **~1.5s** 左右。
- **实际测量**：总耗时 **4.51s**。
- **命令行报错信息**：
  ```
  AssertionError: 熔断降级时延严重累加！总耗时为 4.51s。当 Agora 网格假死时，多个执行步骤会导致严重的系统挂起挂死（每一步都遭遇 1.5s 延时，未熔断）。
  assert 4.511929035186768 < 3.0
  ```
- **分析结论**：每一步都无差别地发起了 HTTP 连接并各自挂起 1.5s。证实 **Swarm 后端完全没有熔断器或状态缓存，在发生超时故障时时延呈线性累加**。若以默认超时 120s 计算，3 个步骤将导致整个工作流挂死 **360 秒**！

### 实验二：Agora 后端探测延迟累加 (agora_mcp_backend.py)
- **模拟故障**：Agora 网关持续假死宕机，健康探针 `client.get(..., timeout=2)` 每次均挂起 2.0 秒超时。对工作流执行 2 次。
- **理论预期**：若具备全局不可达状态的缓存（如 10s TTL 熔断状态），第一次运行等待 2.0s 探测失败后，第二次执行应直接使用熔断器降级状态，耗时趋近于 0s。
- **实际测量**：第一次运行耗时 2.01s，第二次运行仍然耗时 **2.00s**。
- **命令行报错信息**：
  ```
  AssertionError: 缺乏不可达状态的缓存熔断！第二次运行仍然被挂起探测了 2.00s。当 Agora 持续宕机时，每次运行都将增加不必要的网络超时开销。
  assert 2.00222110748291 < 0.2
  ```
- **分析结论**：即使使用了健康探针探测，系统在服务持续宕机时依然在每次执行时无意义地等待 2.0s。在高频或短周期调度的微服务环境中，这 2.0s 的无效探测开销将迅速导致连接池爆满、上游积压乃至 OOM 崩塌，缺乏全局宕机状态的零延迟缓存。

## 3. `test_swarm_no_subprocess.py` 测试严密性评审

我们针对 `m1_worker_1` 提交的 `test_swarm_no_subprocess.py` 进行了严密的对抗性分析，发现其在设计上存在以下严重的“幻觉胜利”缺陷：

### 1) Mock 机制过于理想化 (False Positive Fallback)
- **问题**：测试中使用的 fallback 模拟方式为：
  ```python
  mock_client.post.side_effect = httpx.ConnectError("Gateway connection refused")
  ```
  `ConnectError`（如拒连）在 TCP 握手阶段由系统内核立即返回 RST，所以在 mock 测试下耗时极短（仅为 0.06s），这营造出一种“降级十分迅速”的假象。
- **后果**：该设计直接规避了在网络异常（如网络丢包、握手挂起、服务端线程池满载）引发连接超时时的严重挂死情况。

### 2) 缺乏延迟断言约束 (No Latency Bounds)
- **问题**：测试断言中只检查了工作流是否最终通过降级执行成功 (`passed == 1` 与 subprocess 有调用记录)，**未对执行耗时做任何边界限制**。
- **后果**：即使在真实宕机时因为没有熔断而挂死了整整 120 秒才降级成功，该测试依然会给出 `PASSED` 的绿色标识。这在保障“高可用、零延迟降级”这一核心非功能性指标上形同虚设。

### 3) 未覆盖多步骤的级联效应 (No Cascading Validation)
- **问题**：测试只构造了一个包含单一步骤的工作流。
- **后果**：无法暴露多步骤依次请求挂死、延迟线性相加的真实瓶颈。

### 4) `backends/swarm.py` 中硬编码超时的盲区
- **问题**：测试对 `backends/swarm.py` 中直接使用 `httpx.Client(..., timeout=120.0)` 的硬编码没有任何防护审计。
- **后果**：当 Agora 容器处于僵死状态（连接无响应）时，这会导致灾难性的 2 分钟挂死。

## 4. 优化与防御性建议

为实现真正的“零延迟降级与异常熔断”，建议在 `ecos` 核心层引入以下防护：

1. **引入轻量级全局熔断器 (Circuit Breaker)**
   - 在内存中引入一个全局的 Agora 网格状态缓存（可用一个带 TTL 的全局变量或使用 `cache.py` 统一管理）。
   - 如果发生 `httpx.ConnectTimeout` 或 `ConnectError`，将 Agora 的状态标记为 `UNAVAILABLE`，并配置一个 TTL（例如 30s）。
   - 在 TTL 期间，所有后续的步骤（甚至其他工作流）不再发起任何 HTTP 请求，而是瞬时直接回退到本地 subprocess / mock 降级。
2. **降低首次连接探测超时时间**
   - 将对 Agora 健康探测或 RPC 连接的初始连接超时限制（`connect timeout`）从 120.0s 强制下调为更合理的阈值（如不超过 2.0s，甚至 1.0s），避免因建立连接挂死而阻塞过久。
3. **完善单元测试**
   - 必须引入类似 `test_swarm_backend_no_circuit_breaker_delay_accumulation` 的时间断言用例，确保在假死状态下，多步工作流的运行整体耗时不超过单次探测超时的上限，以此作为降级测试的黄金红线。
