# OMO v4.0 架构元反思：超越四平面与动态演化机制

> **SSOT 状态**：Active  
> **领域**：体系核心理论 (The Constitution)  
> **版本**：v4.0 (Evolutionary Architecture)

## 1. 架构元反思：四平面的局限性

在 OMO 3.0 时代，我们确立了**四平面法则**（Control, Truth, Knowledge, Delivery）。
这是一个非常伟大的“空间整理学”：它完美解决了系统状态的混乱问题，确保了多 Agent 协作时没有“事实脑裂”。

**然而，四平面架构存在一个本质的局限性：它太“静”了。**
四平面本质上是一个**文件系统的映射映射 (File System Overlay)**，它定义了“数据该放在哪里”，但它没有回答：
1. **时间维度**：数据如果过期了、发臭了怎么办？（熵增）
2. **主动性维度**：谁来打扫战场？谁来决定哪些临时代码该转正？
3. **生命周期维度**：系统如何自我生长和愈合？

一个纯粹的四平面架构，最终会变成一座井井有条但堆满落灰档案的“死寂图书馆”。

---

## 2. OMO 4.0 的核心跃升：引入“演化引擎” (The Evolutionary Engine)

为了让系统从“图书馆”变成“生命体”，OMO 4.0 在静态的四平面之上，正式引入了**第五维度 —— 动态演化环 (The Metabolic Loop)**。

如果说四平面 (K0) 是系统的 **DNA（基因图谱）**；
那么演化环 (K1) 就是系统的 **免疫系统与核糖体（执行引擎）**。

### 2.1 动态演化的三大核心机制

#### 机制 A：感知与熵增阻断 (Perception & Anti-Entropy)
系统必须具备“痛觉”。当架构偏离、代码腐化、死锁产生时，系统需要自动标红。
*   **落地形态**：`projects/omo-debt` (Pattern 09 技术债评估系统)。
*   **运作方式**：独立于主线任务之外，巡逻代码库和 `_truth` 平面，动态计算项目的“诚实度”和“技术债跑分”，如果跑分掉入 P0 危险区，则自动强行在 `_control/` 面插入阻断任务。

#### 机制 B：代谢与垃圾回收 (Metabolism & GC)
系统不能只吃不拉。那些执行完的痕迹、死掉的 Agent 记录、废弃的 Draft，必须有机制自动消解。
*   **落地形态**：`omo worker gc` (核心引擎的清理器) 及 `_archive/` 降级平面。
*   **运作方式**：引擎底层会基于时间戳 (`run_stamp`) 和锁 (`locks.db`) 自动判断任务和 Agent 的死活状态，将死寂状态扫入归档区，保证“活跃工作区”永远不超过人脑的认知负荷（<7个）。

#### 机制 C：变异与升维 (Mutation & Growth)
当一个局部规律被反复使用时，它不应该永远躺在 `tasks/` 脚本里，而应该“升维”成系统的基础设施。
*   **落地形态**：联邦分形协议 (Federated Fractal Protocol)。
*   **运作方式**：一旦 `.omo` 实例里的某种通用脚本变得稳定，系统演化机制会将其抽出，放入全局的 `projects/` 目录下（例如本次将 `omo-debt` 剥离出 `.omo`），成为系统级的原生能力，从而实现系统的**自举 (Bootstrapping)**。

## 3. 全景架构：空间与时间的十字交叉

OMO v4.0 的最终形态，是**静态四平面**与**动态演化引擎**的十字交叉验证。

## 4. 落地结论 (Governance Edict)

OMO 系统的设计初衷不再是“不犯错的代码生成器”，而是**“具备强自我修正与演化能力的智能体栖息地”**。

1. **绝对禁令**：禁止在 K0（四平面）内存放任何运行时 Python 脚本，以防阻碍系统的代谢动作。
2. **引擎扩展**：所有的动态能力（计算、巡检、清洗）必须封装为 K1 的独立包（存放在 `projects/`），并通过严格的 CLI 接口读取 `.omo/` 状态。
3. **生死流转**：任何在 `.omo/_knowledge/drafts/` 停留超过 30 天的非活跃提案，必须被代谢系统强制移入 `_archive/`。系统资源只向高优先级的活跃目标倾斜。
