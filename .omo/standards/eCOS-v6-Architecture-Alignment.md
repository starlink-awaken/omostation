# eCOS v6 绝对架构对齐白皮书 (Absolute Architecture Alignment)

> **修订状态**: 最终对齐 (Final Alignment)
> **对齐基准**: 5+4+1+1 (eCOS v6) 与 全链路流程抓手 (The Cybernetic Handles)

在系统从 eCOS v5 向 eCOS v6 的跨越中，我们正式抛弃了零散的功能补丁，将整套“从战略意图到微观代码”的工作流，**严丝合缝地镶嵌进 5+4+1+1 的系统物理坐标系中**。

本文档是对这一大一统架构的终极对齐和客观释义。

---

## 🏛️ 第一维度：空间映射 (5+4+1+1 拓扑矩阵)

### 5 大运行物理分层 (L0-L4) —— 动作在哪里发生？
1. **L4 自我层 (`l4-kernel`)**: 系统意志的最高法庭。存放着 `current.yaml` (CARDS) 与您的北极星愿景。
2. **L3 入口层 (`cockpit` + mounted apps)**: 人机/机机交互的唯一控制面板。`cockpit` 是唯一人类 CLI / Web 入口；`hermes-console`、`dashboard_server` 作为挂载子应用提供界面扩展；`agora-dashboard` 仅保留历史快照，不再作为独立入口。
3. **L2 引擎面 (`omo`, `metaos`, `kairon`, `gbrain`)**: 系统的职能部门。负责治理门控、编排发散、知识记忆的推演。
4. **L1 运行时 (`runtime`)**: 沙箱执行区 (Sandbox)。未经审批的 Pitch 草稿、Agent 正在浴血奋战的测试环境，皆被隔离于此。
5. **L0 协议层 (`ecos`)**: 系统的核心底座。存放所有不可篡改的律法、契约和边界定义。

### 4 大治理约束维 (X1-X4) —— 拿什么标准衡量？
1. **X1 (架构边界与硬约束)**: 控制系统拓扑与合法性。*（例如：Pitch 必须持有 Upstream 锚点，否则拦截）。*
2. **X2 (生命周期与新鲜度)**: 防范系统腐败与垃圾堆积。*（例如：Sandbox 闲置文件 28 天自动触发 GC 销毁）。*
3. **X3 (价值与成本控制)**: 控制资源消耗与 ROI。*（例如：严格执行 Pitch 中设定的 Appetite 预算上限，超支即熔断）。*
4. **X4 (SSOT 与一致性)**: 保证代码与文档不脱节。*（例如：任务完结必须触发经验与代码的向后回写）。*

### 1 层服务织网 (I0) —— 数据怎么流动？
*   **I0 (`agora`)**: 服务网格总线。打碎分层孤岛，强制所有跨层级、跨引擎的调用必须基于 `bos://` 协议，并在该层接受统一的鉴权与准入审查。

### 1 个元模型原点 (M0) —— 世界由什么构成？
*   **M0 (Meta Object Facility Root)**: 万物皆模型的造物起点。所有的流程、文档、卡片，在底层皆服从 M3 (规范) -> M2 (Schema) -> M1 (实例) 的模型驱动哲学，保证系统高度可计算。

## ⚙️ 第二维度：管线穿透 (全链路如何跑通 5+4+1+1)

当您在终端敲下最高层指令 **`workspace iterate "某个宏大构想"`** 时，系统内部发生的“黑盒化学反应”完美穿透了整个 5+4+1+1 矩阵：

| 抓手动作 | 在哪个物理层 (L) 发生？ | 受哪些治理维 (X) 约束？ | 穿透与生效逻辑 |
| :--- | :--- | :--- | :--- |
| **V2P** (意图降维) | **L4 ➡️ L1** | 服从 **M0** 元模型 | 您的构想从 L4 降维，并在 L1 沙箱内生成一个带有边界预期的 M1 实体 (`Pitch.md`)。 |
| **C2G** (门禁下注) | 经由 **L3 ➡️ I0 ➡️ L2** | 受控于 **X1** (架构边界) | OMO (L2) 被唤醒，调用底层 L0 (eCOS) 律法扫描 Pitch。若合规，正式落账 L4 的 CARDS。 |
| **BMAD** (全知铺展) | 发生于 **L2** (Kairon) | 受控于 **X4** (一致性) | Kairon 顺着 `context_uri` 将 L4/L1 的关联知识瞬间拉平，为下一步执行提供无损背景。 |
| **OSC** (约束设计) | 发生于 **L1** (Runtime) | 受控于 **M0/X1** | Agent 依据 M2 Schema 和上游的 No-Gos 禁区，在沙箱内完成最克制的架构蓝图。 |
| **GSD & TTY** (执行与退让) | 发生于 **L1** (Runtime) | 受控于 **X3** (成本上限) | Agent 刺刀见红地写代码。一旦执行开销有刺穿 Appetite 的趋势，X3 律法发威，强制 TTY 退让止损。 |
| **OPC & SWR** (门禁与回写) | 经由 **L1 ➡️ L0 ➡️ L4**| 受控于 **X1 & X4** | 代码须接受 OPC 基线体检 (X1)；通过后，执行残骸被强行聚合并回写入原文档 (X4 SSOT 闭环)。 |
| **AGC** (审计与回收) | **L2** 后台巡逻 | 受控于 **X1 & X2** | OMO 持续扫描大盘，基于 X1 防范战略偏航，基于 X2 清洗 L1 沙箱中的过时垃圾。 |

## 👁️ 最终系统哲学对齐

通过这次全面对齐，我们确认了 **eCOS v6 不是一套散装的脚本集合，而是一台高精密、强自洽的赛博物理引擎。**

*   **人类 (您)** 在 L3 拨动开关，在 L4 眺望远方。
*   **引擎 (OMO/MetaOS/Kairon)** 在 L2 思考与流转。
*   **劳工 (Agent)** 在 L1 的血泥里厮杀。
*   而这艘巨舰之所以不会偏航、不会解体、不会被无休止的 Bug 拖垮，全靠着 **L0 底座上铭刻的 X1-X4 法典** 和 **M0 元模型的绝对约束**。
