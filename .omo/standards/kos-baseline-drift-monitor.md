## 2. Health Check Metrics

### 2.1 Total Docs Drift

```
metric: total_docs_drift
source: kos get_system_status → documents.count
baseline: 7327
current: <live query result>
delta: current - baseline
drift_pct: delta / baseline × 100
threshold: 
  - warning:  ±5%  (367 docs)
  - critical: ±10% (733 docs)
```

### 2.2 Per-Zone Drift

```
metric: per_zone_drift
source: kos get_system_status → zones.{name}.count
zones: [sharedbrain, kairon, gbrain, omo, agentmesh, workspace_root]
threshold:
  - warning:  ±10% change in any single zone
  - critical: ±20% change in any single zone, OR zone missing
```

### 2.3 Missing Zone

```
metric: missing_zone
source: kos get_system_status → zones
action: if any baseline zone is missing → CRITICAL alert → human review
example: sharedbrain zone shows 0 docs → likely reindex failure → DO NOT auto-reindex
```

### 2.4 DB Unavailable

```
metric: db_unavailable
source: try: open(kos-index.sqlite), catch: alert
action: if index file not accessible → CRITICAL alert
note: this usually means process crash or disk issue, NOT reindex problem
```

### 2.5 Known-Doc Retrieval Regression

```
metric: known_doc_retrieval
source: run 10 known doc queries → count success
baseline: 10/10
threshold:
  - warning:  8/10 or 9/10
  - critical: < 8/10
action:
  - warning:  log + notify human
  - critical: CRITICAL alert → DO NOT auto-reindex → human investigates
```

## 3. Drift Threshold Summary

| Metric | Warning | Critical | Auto-action |
|--------|---------|----------|:-----------:|
| total docs drift | ±5% (367 docs) | ±10% (733 docs) | log only |
| per-zone drift | ±10% single zone | ±20% or zone missing | log + alert |
| missing zone | — | any baseline zone = 0 | CRITICAL alert |
| DB unavailable | — | index file inaccessible | CRITICAL alert |
| known-doc regression | 8-9/10 | < 8/10 | CRITICAL alert |

### ⚠️ No-Auto-Reindex Policy

Canonical policy phrase: **no auto reindex**.

**Under no circumstances** does the drift monitor auto-reindex. Reindex is always **L2 human approval**. The drift monitor's role is:

1. Detect and report drift
2. Provide evidence for human decision-making
3. Generate a structured L2 approval request if reindex is needed

```json
// L2 approval request format (generated by drift monitor)
{
  "_confirmed": false,  // ← default false = deny
  "type": "reindex_request",
  "reason": "total_docs_drift=15% > critical_threshold=10%",
  "baseline": 7327,
  "current_count": 6227,
  "zones_with_drift": ["sharedbrain"],
  "evidence": {
    "known_doc_retrieval": "10/10",
    "db_healthy": true
  },
  "recommendation": "Likely KOS reindex needed for sharedbrain zone",
  "requested_action": "kos run_indexer --zones sharedbrain"
}
```

## 6. Key Design Decisions

- **No auto-reindex**: drift monitoring and reindex are separate concerns. Monitor detects, L2 request suggests, human decides.
- **Drift != problem**: a document count change does not necessarily mean index is corrupted. Human must investigate context before reindexing.
- **Known-doc retrieval is the real quality metric**: count drift can be explained by new indexings or deletions. Retrieval regression is the true indicator of index health.
- **Reindex remains L2**: the `_confirmed: false` guardrail is unchanged and applies to all reindex operations.
- **Sensitive capabilities remain blocked**: this monitor reads health data (L0) and generates reindex requests (L2). It does not touch Apple/WeChat/Family OS/SMB/Media.
