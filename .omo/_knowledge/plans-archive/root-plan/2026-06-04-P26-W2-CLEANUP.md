# OMO Phase 26 Wave 2: 四平面职责清理与重组 (硬对齐方案)

## Background & Motivation (背景与动机)
当前 OMO 治理操作系统处于 Phase 26 Wave 1（控制面同步已完成）。根据 `goals/current.yaml` 的规划，Wave 2 的核心目标是“四平面职责清理”，KPI 要求实现 `_control / _truth / _knowledge / _delivery` 目录的物理边界清晰，无跨平面职责混杂。
此前系统采用的是“导航壳（软链接）”模式，导致 `.omo/` 根目录下堆积了大量的多平面混合业务目录（如 `workers/`, `debt/`）以及过时的墓碑文件，这增加了 Agent 在上下文中检索的熵值，容易引发状态漂移。为了彻底实现架构的高内聚低耦合，我们决定采用**“硬对齐（严格物理隔离）”**策略进行重构。

## Scope & Impact (范围与影响)
本计划将对 `/Users/xiamingxing/Workspace/.omo/` 目录进行大规模的物理重构：
1. **根目录瘦身**：清理废弃的 `.md` 墓碑文件。
2. **域（Domain）拆分**：打破原有的 `workers/`、`debt/` 混合目录，按数据属性分别下沉到 `_truth` (注册表) 和 `_delivery` (运行日志)、`_control` (仪表盘)。
3. **知识面归位**：将散落在根目录的 `diagrams/`, `drafts/`, `task-prompts/`, `scenarios/`, `plans/`, `audits/`, `summaries/` 按分类移入 `_knowledge/` 的对应子目录。
4. **交付面归位**：将 `tests/`, `evidence/` 移入 `_delivery/`。
5. **索引修复**：更新所有相关文件的内部指针。

该改动将触及 `.omo/` 下约 50% 的顶级结构，需要保证过程中的状态一致性，防止断链。

## Proposed Solution (实施方案)

### 步骤 1: 清理根目录墓碑文件
将以下文件彻底移入 `.omo/_archive/`（如果目标文件已存在则覆盖或备份）：
- `AUDIT.md`, `CLEANUP.md`, `CONSISTENCY-CHECK.md`, `GOVERNANCE_PLAN.md`, `HEALTH_DASHBOARD.md`, `ONBOARDING.md`, `P1_PROJECT_HEALTH.md`, `STATE.md`, `TASK_POOL.md`

### 步骤 2: 知识面 (Knowledge Plane) 物理归位
将根目录下的认知类资产按分类移入 `_knowledge/`：
- `mv .omo/plans/ .omo/_knowledge/design/plans/`
- `mv .omo/diagrams/ .omo/_knowledge/reference/diagrams/`
- `mv .omo/task-prompts/ .omo/_knowledge/usage/task-prompts/`
- `mv .omo/scenarios/ .omo/_knowledge/reference/scenarios/`
- `mv .omo/audits/ .omo/_knowledge/management/audits/`
- `mv .omo/drafts/ .omo/_knowledge/management/drafts/`
- `mv .omo/summaries/ .omo/_knowledge/process/summaries/`

### 步骤 3: 交付面 (Delivery Plane) 物理归位
将根目录下的运行证据类资产移入 `_delivery/`：
- `mv .omo/tests/ .omo/_delivery/tests/`
- `mv .omo/evidence/ .omo/_delivery/evidence/`

### 步骤 4: 跨平面业务域拆分 (workers/ 与 debt/)
- **Workers 拆分**: 
  - 运行记录属于交付面：`mv .omo/workers/runs/ .omo/_delivery/workers-runs/`
  - 注册表属于事实面：`mv .omo/workers/registry.yaml .omo/_truth/registry/workers.yaml`
  - 移除空目录：`rm -rf .omo/workers/`
- **Debt 拆分**: 
  - 注册表属于事实面：`mv .omo/debt/registry.yaml .omo/_truth/registry/debt.yaml`
  - 仪表盘属于控制面：`mv .omo/debt/dashboard/ .omo/_control/debt-dashboard/`
  - 报告和审查（如 reporting/ 和 reviews/）移入对应的 `_knowledge/management/` 或 `_delivery/`。
  - 移除空目录：`rm -rf .omo/debt/`

### 步骤 5: 更新关联指针与索引
- 更新 `state/system.yaml` 中可能存在的绝对路径指向（如 `debt_registry_ref` 等）。
- 刷新各平面的 `INDEX.md` (`_control`, `_truth`, `_knowledge`, `_delivery`)，确保所有相对路径链接正确。
- 更新 `.omo/INDEX.md` 根索引。

## Alternatives Considered (替代方案评估)
**软对齐（导航壳模式）**：通过保留原有的物理目录（如 `workers/`, `plans/`），只修改四个平面的 `INDEX.md` 来软连接这些目录。
*拒绝原因*：用户明确选择了“硬对齐（严格物理隔离）”。虽然软对齐破坏性小，但长远看违背了高内聚的治理理念，无法彻底解决跨平面职责混杂的 KPI 要求。

## Verification (验证标准)
1. 运行 `ls -l .omo/`，根目录下除四个带 `_` 的平层目录、`goals/`, `tasks/`, `state/`, `standards/` 和基础配置文件外，不应存在多余的散落目录（如 `diagrams`, `evidence`）和废弃 `.md` 文件。
2. 验证各个平面的 `INDEX.md` 内部链接，无 404 死链。
3. 检查 `.omo/state/system.yaml` 中记录的引用路径均已重定向到新位置。

## Migration & Rollback (迁移与回滚策略)
**回滚策略**：在执行任何 `mv` 或 `rm` 动作前，先对当前的 `.omo/` 目录进行完整的 `cp -r .omo .omo.bak_p26w1` 备份。如果在指针修改中发生大面积破损或与上层脚本出现不可解决的冲突，通过 `rm -rf .omo && mv .omo.bak_p26w1 .omo` 一键回滚。
