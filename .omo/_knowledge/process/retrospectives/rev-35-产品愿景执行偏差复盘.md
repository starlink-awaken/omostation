# 产品愿景执行偏差复盘

> 日期: 2026-05-26
> 事件: PRODUCT_VISION.md (2026-05-23) → 3 天后 90% 精力投入架构治理,产品端零推进

---

## 一、根因链

```
表象: 3天, 90% 治理, 10% 产品
  ↓
直接原因: AAMF 审计 (发现1-5) → Phase 1-11 自然延伸 → 无停检点
  ↓
结构原因: 产品愿景仅 3 天历史, 未内化为工作流门禁
  ↓
深层原因: 治理任务可全自主完成 (脚本/配置/文档), 产品任务需用户反馈 (UX/设计/验证)
  ↓
根本原因: 系统无"产品偏置"机制 — 所有门禁都来自工程/架构视角, 产品视角无声
```

### 根因1：治理是"低反馈路径"

| 维度 | 治理任务 | 产品任务 |
|------|---------|---------|
| 能否自主完成 | ✅ 全自主（写脚本/配文档） | ❌ 需要用户决策（UX/流程/验证） |
| 反馈周期 | 即时（跑一次就知结果） | 数小时到数天（需要用户试用） |
| 明确性 | 高（规则明确, 对错清晰） | 低（"感觉对不对"需要判断） |
| **Agent 自然倾向** | 🟢 有产出就继续 | 🟠 遇到模糊就停滞 |

Agent 天然倾向"可自主完成的任务",治理恰好就是这种任务。

### 根因2：无"停检点"机制

AAMF 审计发现 5 个问题 → Phase 1 方案 → Phase 2 → ... → Phase 11。每个 Phase 完成时问的都是"继续推还是换方向?",没有问"这个方向还和产品愿景一致吗?"。

```
Phase 边界问题: "继续推?"
没有问: "这推进的是产品还是治理?"
```

### 根因3：产品视角不在工作流中

AGENTS.md、项目上下文、系统提示词全部从工程/架构角度描述项目。没有任何地方写着"这个改动是否符合产品愿景？"。

```
工作流门禁:
  ✅ 代码编译
  ✅ 测试通过
  ✅ 架构合规
  ❌ 产品愿景对齐
```

---

## 二、预防机制

### 机制1：产品门禁 — 每次任务前强制对齐

在 AGENTS.md 中增加：

```
## 产品门禁
每次开始新任务前,必须回答:
1. 这个任务服务于哪条产品原则? (①-⑤)
2. 它让用户哪条路径更顺滑?
3. 如果答案是"服务治理/架构",那它最终如何体现在产品端?
```

### 机制2：精力分配硬约束

```
单次迭代:
  ≤ 30% 基础设施/治理/架构
  ≥ 70% 产品功能/用户旅程/体验
```

超过 30% 治理任务 → 必须先完成一条用户旅程才能继续。

### 机制3：Agent 记忆植入 5 条原则

将 5 条产品原则写入 Agent 持久记忆,每次对话自动加载,无需翻文档。

### 机制4：Phase 完成检查清单扩充

```
当前 Phase 完成检查:
☐ 代码通过
☐ 测试通过
☐ 架构合规

新增:
☐ 产品愿景对齐 — 本 Phase 改进了哪条用户旅程?
☐ 精力分配回顾 — 治理/产品比例是否合理?
```
