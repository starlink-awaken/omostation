# 目标纠偏审计：Workspace CLI vs 4+1+3 分层架构

> 审计日期: 2026-05-26
> 审计范围: workspace CLI (Phase 0-4) 与 `09-个人AI操作系统-最终架构方案.md` 定义的 4+1+3 分层架构方向一致性
> 审计方法: 逐层对标、能力覆盖分析、缺口识别

---

## 一、核心问题：Workspace CLI 在 4+1+3 中处于什么位置？

**结论：Workspace CLI 是一个跨层产品界面，不是 4+1+3 中的独立层。**

4+1+3 架构定义的是系统内部结构（"系统由什么组成"），
Workspace CLI 定义的是用户交互界面（"用户用什么入口"）。

两者不是替代关系，是**互补关系**：

```
┌─────────────────────────────────────┐
│  产品界面                            │
│  L4→L3→L2 用户入口                  │
│  workspace CLI ← 这里              │
├─────────────────────────────────────┤
│  4+1+3 系统架构                      │
│  L4 自我 | L3 协作 | L2 能力 | L1 契约│
│  X1 治理 | X2 抗熵 | X3 价值堆栈     │
└─────────────────────────────────────┘
```

---

## 二、逐层对标

### L4: 自我层 — "我是谁、为什么"

| 要求 | 现状 | 评价 |
|------|------|------|
| 身份画像 | ❌ 无 | 🔴 |
| 愿景系统 | ❌ 无 | 🔴 |
| 价值原则 | ⚠️ AGENTS.md 有产品原则（Agent 级）但 workspace CLI 未体现 | 🟠 |
| 认知框架 | ❌ workspace CLI 未展示任何框架 | 🔴 |
| 交付档案 | ⚠️ 文档在 `~/Documents/学习进化/` 但 CLI 无入口 | 🟠 |

**分析**: workspace CLI 完全缺失 L4 层能力。同一个用户可能在 Hermes Agent (MBP) 中定义身份原则，但 workspace CLI (Mac mini) 对此一无所知。**这是最大的方向偏差**。

**修复建议**: `workspace profile` 或 `workspace identify` 子命令，链接到身份档案。

### L3: 协作层 — "怎么做、谁来做"

| 要求 | 现状 | 评价 |
|------|------|------|
| 共享工作平面 | ⚠️ 研究记录是跨对象的但无多Agent协作 | 🟠 |
| 多Agent接入 | ❌ CLI 是单用户单终端 | 🔴 |
| Human Node | ⚠️ 通过 CLI 输出但无双向交互 | 🟡 |

**分析**: workspace CLI 目前是纯单人工具。4+1+3 设想的 L3 是多 Agent 在共享工作平面上协作。workspace CLI 的研究对象本身可以作共享工作平面，但缺少多 Agent 接入点。

**修复建议**: 在研究对象层支持多 Agent 跟踪（`--agent <name>`），或者通过 Agora 注册 workspace 操作作为 MCP 能力。

### L2: 能力层 — "用什么做"

| 要求 | 现状 | 评价 |
|------|------|------|
| Agora (MCP路由) | ⚠️ workspace status 检测 CLI 可达但未调用 | 🟡 |
| KOS (知识索引) | ⚠️ 推测有存储但无直接对接 | 🟡 |
| minerva (研究) | ⚠️ research 命令 delegate 到 minerva CLI 或降级回复 | 🟡 |
| Forge (工具图谱) | ❌ 未对接 | 🔴 |
| gbrain (记忆) | ❌ 未对接 | 🔴 |
| kronos (摄取) | ⚠️ import 命令覆盖了部分类似功能 | 🟡 |

**分析**: workspace CLI 与 L2 能力层的对接是"片段式"的 — 部分 CLI 检测、部分进程调用、大部分自实现。**没有统一的 MCP 调用路径**。

**修复建议**: 将 workspace CLI 的 backend 操作逐步迁移到 Agora MCP 调用，而非本地直接 SQLite。

### L1: 契约层 — "什么格式"

| 要求 | 现状 | 评价 |
|------|------|------|
| WorkspaceObject Schema | ✅ workspace contracts 系列命令完整 | 🟢 |
| IdentityEnvelope | ❌ 未实现 | 🔴 |
| EventEnvelope | ⚠️ timeline 追踪了事件但非标准格式 | 🟡 |
| Principle/Decision | ❌ 未实现 | 🔴 |

**分析**: 契约层是 workspace CLI 唯一做得好的层。`contracts validate/list/export-research` 直接使用 Eidos/SSOT 的 Schema，方向正确。

**修复建议**: 保持并扩展。增加 `contracts export --identity` 和 `contracts export --event`。

## 三、横切维度对标

### X1: 治理安全

| 要求 | 现状 | 评价 |
|------|------|------|
| 身份/授权 | ❌ 无 | 🔴 |
| 审计日志 | ✅ workspace governance 委派 arcnode-* 有 SHA256+GPG | 🟢 |
| 免疫模型 | ❌ 无 | 🔴 |

### X2: 抗熵与进化

| 要求 | 现状 | 评价 |
|------|------|------|
| 保鲜策略 | ❌ 无自动保鲜 | 🔴 |
| 轻量复盘 | ⚠️ daily 可作复盘但无自动触发 | 🟡 |
| 自回收 | ❌ 无 | 🔴 |

### X3: 价值堆栈

| 要求 | 现状 | 评价 |
|------|------|------|
| 价值层次 | ❌ 无 | 🔴 |
| 半衰期 | ❌ 无 | 🔴 |
| 引用链 | ⚠️ dossier 可追踪关系但无半衰期计算 | 🟡 |

**分析**: 横切维度几乎全部缺失。workspace CLI 目前只覆盖了 X1 的审计日志部分。**治理安全层是完整程度最低的横切维度**。

## 四、一致性评分

| 层 | 权重 | 评分 | 加权分 |
|----|------|------|--------|
| L4 自我层 | 20% | 10% | 2% |
| L3 协作层 | 20% | 15% | 3% |
| L2 能力层 | 25% | 35% | 8.75% |
| L1 契约层 | 15% | 75% | 11.25% |
| X1 治理安全 | 10% | 25% | 2.5% |
| X2 抗熵 | 5% | 10% | 0.5% |
| X3 价值堆栈 | 5% | 10% | 0.5% |
| **总计** | **100%** | — | **28.5%** |

**综合评分: 28.5/100 🔴 — 方向一致但严重偏科**

workspace CLI 主要覆盖了 4+1+3 中的 L2 能力层和 L1 契约层，但 L4 自我层、L3 协作层、以及所有横切维度几乎完全缺失。

## 五、关键偏差总结

| 偏差 | 来源 | 影响 |
|------|------|------|
| workspace CLI 在架构中没有定义位置 | 架构设计遗漏 | 🔴 导致 L4/L3 完全空缺 |
| 能力层对接是"片段式"而非 MCP 统一 | 实现方式选择 | 🟡 冗余实现增加维护成本 |
| 横切维度几乎全无 | 产品需求优先级 | 🔴 抗熵闭环不能自动运行 |
| L4 自我层完全缺位 | 架构分层未对齐 | 🟠 系统不认识"我是谁" |

## 六、修复建议

### 短期（下一迭代）

1. **添加 `workspace profile`**: 链接到身份档案（L4 入口），显示当前身份/角色/原则
2. **添加 `workspace contracts export --identity`**: 补全契约层缺失的 IdentityEnvelope
3. **Daily 增加保鲜提示**: 对超过 3 天未操作的活跃研究自动提醒归档（X2 基础）

### 中期

1. **Backend 迁移到 Agora MCP**: workspace CLI 的研究存储从本地 SQLite 迁移到 Agora/KOS 的 MCP 调用（L2 标准化）
2. **支持 `--agent <name>`**: 在研究对象上标记处理 Agent，为多 Agent 协作铺垫（L3 基础）
3. **半衰期计算**: 按最后活跃时间计算研究对象价值衰减（X3 基础）

### 长期

1. **workspace CLI 作为 Agora MCP 的客户端**: 所有操作通过 Agora MCP 调用
2. **多用户协作**: 将研究对象共享到 Agora 的共享工作平面

---

## 七、结论

Workspace CLI 与 4+1+3 分层架构的方向**部分一致**（28.5/100）。核心问题不是"方向错了"，而是"只做了其中两层的部分功能"。

**最大的矛盾**：之前 90% 的精力花在治理上（X1），偏离了产品方向。但此轮修复（Phase 0-4）把精力放到产品入口上（workspace CLI），又偏离了分层架构中 L4/L3/X2/X3 的覆盖。产品推进和架构覆盖之间需要找到平衡点。

**当前阶段正确的事**：从用户视角建立产品入口（workspace CLI）而不是从系统架构视角推功能，符合"产品优先"原则。等产品基础牢固后再逐层补齐缺失的架构层。
