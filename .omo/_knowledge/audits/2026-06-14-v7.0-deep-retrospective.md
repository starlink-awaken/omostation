# eCOS v7.0 深度复盘与债务治理报告 (Deep Retrospective)

> 状态: active | 阶段: v7.0 纪元首日 | 关联基线: `v6.1-strategic-execution-roadmap.md`

## 🎯 一、 核心复盘语境 (Context)

随着 Phase 4 的顺利收口，eCOS 体系完成了从 v6.1 的 **单体混沌** 向 v7.0 的 **机制驱动 (Governance-Driven) 与 混合防腐** 的跨越。我们在 Agora 层实装了 RBAC、MetaOS Admission 准入，并在 Kairon 观测域打通了 VibeOps。

**但“破窗”之后的系统，伴生了新的拓扑断层与技术债务。**
按照 SharedBrain `B.D.S.K.` 的工作原则，本次复盘结合 **模型驱动 (Model-Driven)** 的 M1↔M0 对齐，并严格套用 **X1-X4 治理维度**，直面系统的隐患与伤疤。

---

## 🏚️ 二、 债务清册与 X1-X4 治理映射 (Debt Matrix)

经过 `make kairon-test` 与 `agora` 域内总计 4000+ 测试案例的实弹洗礼，系统暴露了以下 6 项核心债务（按破坏力 P0-P3 排序）：

### Debt 01: [P0] 第三方生态路由全面瘫痪风险
* **现象 (M0 Reality)**: `agora` 的 `test_github_source_e2e.py` 等 24 个非核心集成测试全线失败。
* **根因**: 我们在 `ProxyManagerDynamic` 实施了最严苛的 `metaos_admission` 白名单拦截。非 Core 服务（如 github-source）因未获得准入票据被直接抛弃。
* **X1 约束 (硬性边界)**: `CR-ADMISSION-01` 发挥了正面作用，成功拦截。
* **X4 约束 (一致性)**: 违反 `CS-10: BOS URI 溯源一致性`。新增插件缺乏标准的声明式注册与准入审批流程。
* **治理决议**: 开发 **插件注册模型 (Plugin Manifest Schema)**，所有非核心生态必须在 M1 模型层提交 `yaml` 声明，经 MetaOS 审计后自动下发准入 JWT。

### Debt 02: [P1] Cockpit 终端 Token 断层
* **现象 (M0 Reality)**: 命令行入口目前缺乏真正的鉴权流传，只能靠硬编码或忽略机制度日。
* **根因**: 底层 Agora 已全面实行基于 JWT `RoleContext` 的 `CR-RBAC-01` 检验，但 L3 `cockpit` 尚未接入任何身份提供商 (IdP)。
* **X1 约束 (边界)**: `CR-RBAC-01` (Generator 无权访问 Evaluator)。
* **治理决议**: 必须在 `cockpit` 引入类似于 Auth0 或本地化的 PKI 密钥环，在执行 `cockpit ...` 时动态签名注入 `Authorization: Bearer <token>`。

### Debt 03: [P1] AgentRuntime 历史遗迹导致测试脏乱
* **现象 (M0 Reality)**: `test_degrade.py` 及跨域链路大量报错 `ModuleNotFoundError` 或 SKIP。
* **根因**: 系统架构升维后，早期的 `agent-runtime` 被暴力废除移至 `_archived`，但测试桩依然存留。
* **X4 约束 (一致性)**: 严重违反 `CS-02` (设计与开发对齐)。M0 代码已删，M1 契约 (测试) 未删。
* **治理决议**: 发起为期半天的 **Tech Debt 清理冲刺**，彻底将历史组件从 AST 语法树和测试夹具中抹除。

### Debt 04: [P2] 跨仓测试边界渗透
* **现象 (M0 Reality)**: `test_vibeops_analysis_integration.py` 此前被放错了位置（放在 agora 但引用 kairon），引发环境崩溃。
* **根因**: Monorepo 的 `uv` workspace 虽然扁平，但 Python 的 `sys.path` 依然具备物理隔离性。
* **X4 约束 (一致性)**: 缺乏跨域依赖图 (DAG) 的 lint 校验。
* **治理决议**: 在 GitHub Actions CI 或 `make governance-verify` 中增加 `import-linter`，强制阻断 `agora` 导入 `kairon_observability`。

### Debt 05: [P2] A2A 协议虚空楼阁
* **现象**: 出具了 A2A (Agent-to-Agent) 可行性报告，但物理层依然是单进程 `asyncio.Queue`。
* **X2 策略 (阈值告警)**: 单节点承载力已达上限，一旦有长程推演 Agent 并发，即会触发 `alert_threshold` 熔断。
* **治理决议**: 按 roadmap 延后至 v7.1 阶段实施，引入 NATS JetStream 替换。

### Debt 06: [P3] 混合验证 (LLM-as-Judge) 的吞吐黑洞
* **现象**: DeepEval 的 `FaithfulnessMetric` 等高阶验证严重依赖大模型并发。
* **X3 价值 (成本归因)**: API 调用激增，且偶发触发 Rate Limit。若测试频次过高（如每次 PR 都跑），会拉垮 X3 定义的 ROI 阈值。
* **治理决议**: 针对 `@pytest.mark.vibeops` 打标的用例，实施 **降采样 (Downsampling) 策略**，日常 CI 只测 10%，Nightly 测 100%。

## 🛠️ 三、 模型驱动校验 (Model-Driven Validation)

对照 `ecos/ssot/mof/m1/governance/` 模型，本次复盘得出系统级结论：

1. **强一致性胜利 (Strong Enforcement)**: `CR-RBAC-01` 与 `CR-ADMISSION-01` 作为 X1 防火墙已通过实战检验，甚至将我们自己的旧测试 (github-source) 成功拦截。这证明 L0 模型不是“写在纸上的规范”，而是物理运行时的“法典”。
2. **模型漂移警告 (Model Drift)**: `agora` 中 24 个失败的测试代表了典型的 **实现层 (M0) 抛弃了模型层 (M1)**。我们在删减功能时，未对 M1 模型树进行 `Cascading Delete`。

## 📅 四、 Next Action (行动倡议)

若要让 v7.0 稳固扎根，我们需要开启一个小型的 **“大扫除 Sprint” (v7.0.1 Debt Cleanup)**：
1. **[任务 1]** 实施 `cockpit` CLI 的 JWT 生成伪装机制，打通 L3 -> I0 的全真鉴权链路。
2. **[任务 2]** 清理所有涉及 `agent-runtime` 的坏死代码和失效测试用例。
3. **[任务 3]** 为非核心插件（如 `github-source`）增设 `MetaOS` 准入清单白名单映射。

**系统架构师总结**：
> "真正的架构不是不犯错，而是能**敏锐地感知并捕获错误**。v7.0 最大的成功，在于系统自身能精准地暴露这些债务（通过红线拦截和测试熔断），而不是让它们烂在堆栈深处。"
