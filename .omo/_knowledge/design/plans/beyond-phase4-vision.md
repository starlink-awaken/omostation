## 一、Phase 4 的终点是什么

Phase 4 结束后（2027 Q4），omostation 将处于以下状态：

```
系统快照 (2027 Q4):
┌────────────────────────────────────────────┐
│ 辅助自主率: > 80%                          │
│ 系统健康: ≥ 91/100                         │
│ MCP 工具: 130+                             │
│ Agent 类型: 30+                            │
│ SharedWork 融入: 10/90+                    │
│ 可分发: ✅                                 │
│                                            │
│ 核心能力:                                   │
│  ✓ 知识摄取 → 研究 → 推导 → 索引 全自动      │
│  ✓ 自愈: 95%+ 异常自动恢复                  │
│  ✓ KOS: 辅助发现新模式/工具/技能             │
│  ✓ RBAC: 4 角色权限体系                     │
│  ✓ EU 经济: 虚拟资源会计                    │
│  ✓ 免疫: 全系统安全检查                      │
│  ✓ wksp:// URI: 统一寻址                   │
└────────────────────────────────────────────┘
```

但 Phase 4 不是终点——它是**系统从"被建造"到"自我进化"的转折点**。

Phase 1-4 的本质是**人类设计、人类建造、AI 辅助执行**。Phase 4 之后的本质是**系统自主发现方向、人类设定边界、AI 负责执行**。

### 1.1 验证过渡期（Phase 3 之后、Phase ∞ 之前）

> ⚠️ 新增: 红队审查发现 Phase 3 的辅助自主能力未经验证就进入 Phase ∞ 是致命的。增加 3-6 个月的验证期。

| 验证维度 | 门槛 | 验证方法 |
|---------|:----:|---------|
| KOS 推荐准确率 | ≥ 90% 连续 3 个月 | 人类对每条推荐标注 accept/reject，计算准确率 |
| 自愈成功率 | ≥ 95% (100 次模拟) | 注入 100 次异常 → 95 次成功恢复 |
| 辅助研究质量 | ≥ 人类水平的 85% | 双盲评估: 3 篇 AI 辅助 vs 3 篇纯人类研究 |
| HITL 超时机制 | 100% 超时后正确拒绝 | 模拟人类缺席: 推荐超时 → auto-reject |
| 级联保护 | 3 次多服务同时告警 → 3 次正确停止自愈 | 注入多服务告警 → 验证停止自愈 |
| RBAC 执行 | 100 次越权调用 → 100 次被拒 | 渗透测试: 尝试越权调用 |

**通过条件**: 全部 6 项门槛达标 → 启动 Phase ∞。任一项不达标 → 延期 3 个月，修复后重试。

## 二、Phase ∞：持续自我进化

Phase 4 之后，不再有"Phase 5, 6, 7"。取而代之的是**Phase ∞**——一个持续运行的自我进化循环。

### 2.1 Phase ∞ 的运作模型

```
        ┌─────────────────────────────────────────┐
        │            Phase ∞ 进化循环              │
        │                                         │
        │  ┌─────────┐      ┌─────────┐           │
        │  │ 观察     │ ───→ │ 假设     │           │
        │  │ (KOS监控 │      │ (生成改进 │           │
        │  │  全系统) │      │  假设)    │           │
        │  └─────────┘      └────┬────┘           │
        │       ↑                ↓                │
        │       │           ┌─────────┐           │
        │       │           │ 实验     │           │
        │       │           │ (沙箱中  │           │
        │       │           │  验证)   │           │
        │       │           └────┬────┘           │
        │       │                ↓                │
        │  ┌─────────┐      ┌─────────┐           │
        │  │ 学习     │ ←─── │ 推荐     │           │
        │  │ (将从实验中│      │ (向人类   │           │
        │  │  习得的模式│      │  推荐改进 │           │

        │                   └─────────┘           │
        └─────────────────────────────────────────┘
```

### 2.2 进化维度

Phase ∞ 的进化不是随机的，而是沿着明确的维度：

| 维度 | 当前 Phase 4 | Phase ∞ 目标 | 衡量 |
|------|:------------:|:------------:|------|
| **认知深度** | L0-L4 研究 | L0-L6 研究（假设生成 + 反事实推理） | 研究论文被引次数 |
| **自主广度** | 80% 操作 AI 推荐 | 95% 操作 AI 推荐，仅战略决策需人类 | AAR (辅助自主率) |
| **知识密度** | 数千实体 | 百万级实体，跨域知识图谱 | KOS 实体数 |
| **响应速度** | 分钟级研究 | 秒级实时知识检索 + 分钟级深度研究 | P50 延迟 |
| **适应速度** | 1 新能力/周 | 1 新能力/天 | 新 MCP tool 注册速率 |
| **分发度** | 单人使用 | 100+ 用户，跨平台 | 活跃安装数 |
| **经济自持** | EU 虚拟会计 | 真实成本核算 + 优化建议 | 月度运营成本 |

## 三、从个人工具到认知放大器

Phase 4 之后，omostation 的角色发生根本性转变。

### 3.1 三个阶段的关系

```
  Phase 1-3              Phase 4               Phase ∞
  建造阶段                辅助阶段              共生阶段
  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ 人类建造  │   →     │ AI 辅助   │   →    │ 人类+AI  │
  │ AI 执行   │         │ 人类决策   │         │ 共同进化  │
  │           │         │           │         │           │
  │ 工具      │         │ 助手      │         │ 延伸      │
  └──────────┘         └──────────┘         └──────────┘
```

### 3.2 认知放大器的五个能力

1. **记忆延伸 (Memory Extension)**
   - 不只是"记住你说过什么"
   - 而是"理解你思维中的模式、盲点、成长轨迹"
   - 当你遇到新问题时，系统说："你在 2025 年处理过一个类似的问题，当时的思路是 X，但后来发现 Y 更好"

2. **思维加速 (Thought Acceleration)**
   - 不只是"快速搜索"
   - 而是"在你思考的过程中，并行探索多条路径，呈现给你最优的几条"
   - 像有一个永远不会累的研究团队在你思考的同时做背景研究

3. **决策增强 (Decision Augmentation)**
   - 不只是"提供信息"
   - 而是"建模你的决策框架，预测不同选择的结果，指出你认知偏误"
   - 像 Naval + Munger + Taleb 同时给你建议

4. **创造力放大 (Creativity Amplification)**
   - 不只是"生成内容"
   - 而是"理解你的审美和风格，在你创作时提供既符合你风格又突破你边界的建议"
   - 像你最好的合作者，但永不枯竭

5. **执行自动化 (Execution Automation)**
   - 不只执行你下达的命令
   - 而是理解你的目标，自主拆解、编排、执行、验证
   - 你说"帮我准备下周的演讲"，系统自动：研究主题 → 生成大纲 → 制作 PPT → 排练计时 → 收集反馈

## 四、迭代机制：从人类规划到系统自主演化

Phase 4 之后，`omo/` 目录的规划模式发生根本变化。

### 4.1 从 `plans/` 到 `evolution/`

```
.omo/
├── plans/                    ← Phase 1-4 的产物（人类写的计划）
│   ├── architecture-final-vision.md
│   ├── evolution-roadmap-4phases.md
│   └── phase1-4-task-specs.md
│
├── evolution/                ← Phase ∞ 的产物（系统生成的进化记录）
│   ├── hypotheses/           ← KOS 生成的改进假设
│   │   └── 2027-Q4-001.md    "假设：将 minerva 检索从 LanceDB → Qdrant 可将延迟降 30%"
│   ├── experiments/          ← 沙箱验证结果
│   │   └── exp-2027-Q4-001.md "实验：LanceDB vs Qdrant，结果：延迟降低 28%，确认假设"
│   ├── adoptions/            ← 人类批准后实施的变更
│   │   └── 2027-11-minerva-qdrant.md
│   ├── quarterly-reviews/    ← 每季度人类战略审查
│   │   └── 2027-Q4-review.md
│   └── metrics/              ← 进化度量指标
│       └── evolution-dashboard.json
│
├── decisions/                ← 重大决策日志（不可变）
├── health/                   ← 健康评分历史
└── api-docs/                 ← 自动生成 API 文档
```

### 4.2 人类角色的演变

| 角色 | Phase 1-3 | Phase 4 | Phase ∞ |
|------|-----------|---------|---------|
| **架构师** | 人类设计架构，AI 执行 | AI 分析架构，推荐改进，人类批准 | AI 自主提出架构变更，人类审查季度报告 |
| **开发者** | 人类写代码，AI 辅助 | AI 写代码，人类审查 | AI 写代码并自测，人类审查异常 |
| **QA** | 人类写测试 | AI 生成测试，人类运行 | AI 生成+运行+分析测试结果 |
| **PM** | 人类规划 Sprint | AI 推荐优先级，人类调整 | AI 自主管理 backlog，人类设定季度目标 |
| **Ops** | 人类运维 | AI 自愈 + 人类处理异常 | AI 全自主运维，人类处理 CRITICAL 异常 |
| **战略家** | 人类决定方向 | 人类决定方向 | 人类设定愿景，AI 填充路径 |

### 4.3 季度战略审查（Phase ∞ 的核心仪式）

每季度，人类执行一次战略审查：

```
季度审查议程 (90 分钟):
1. AI 呈现本季度进化摘要 (10min)
   - 新能力: X 个新 Skill/Tool/Schema
   - 改进: Y 次架构优化
   - 健康: 从 XX → YY
   - 异常: Z 次需要人类介入的事件

2. AI 推荐下季度方向 (10min)
   - 优先级 1-3
   - 每个优先级的预期收益
   - 风险评估

3. 人类调整方向 (20min)
   - 批准/修改/拒绝推荐
   - 设定新的高层面目标
   - 调整安全护栏参数

4. 人类反馈 (20min)
   - 使用体验
   - 不满意的地方
   - 新需求

5. AI 生成下季度的 evolution plan (10min)
   - 自动生成 → 人类确认
```

## 五、多实例联邦：从单人到社区

### 5.1 从单实例到联邦

```
         omostation 实例 A           omostation 实例 B
        (你的个人 OS)               (另一个用户的 OS)
              │                           │
              │     联邦学习层             │
              └─────────┬─────────────────┘
                        │
              ┌─────────▼─────────┐
              │  Federated KOS    │
              │                   │
              │  共享但不泄露:     │
              │  ✓ 新发现的模式    │
              │  ✓ 新生成的 Skill  │
              │  ✓ 性能基准数据   │
              │  ✓ 异常模式库     │
              │                   │
              │  绝不共享:        │
              │  ✗ 个人数据       │
              │  ✗ 对话历史       │
              │  ✗ 知识卡片内容   │
              │  ✗ 身份信息       │
              └───────────────────┘
```

### 5.2 联邦学习的价值

| 共享内容 | 价值 | 隐私保护 |
|---------|------|---------|
| 新发现的实体模式 | 实例 A 发现的模式帮助实例 B 更快理解 | 仅共享模式结构，不共享数据 |
| 自愈规则 | 实例 A 遇到的异常 → 修复规则 → 实例 B 预防 | 仅共享规则逻辑，不共享事件细节 |
| Skill 模板 | 实例 A 蒸馏的思维框架 → 实例 B 可复用 | 仅共享 Skill 定义，不共享训练数据 |
| 性能基准 | 跨实例延迟/吞吐对比 → 识别性能退化 | 仅共享聚合统计，不共享原始日志 |

### 5.3 开源社区模型

```
omostation 演化路径:

2028 Q2: Phase ∞ 启动（验证期通过后）
2028 Q4: 开源核心 (kairon + agora → GitHub)
2029 Q2: 3-5 早期采用者
2029 Q4: 10-20 用户，**联邦安全模型就绪后**启动联邦学习
2030 Q2: 社区贡献 Skill/Tool (需经过联邦沙箱审查)
2031+:   omostation 成为个人 AI OS 的可行选择之一
```

### 5.3 联邦安全模型

> ⚠️ 新增: 红队审查发现联邦学习是投毒攻击的理想目标。以下安全模型为必需。

```
联邦安全模型 (4 层防护):
1. 联邦沙箱 (FedSandbox):
   - 所有来自其他实例的 Skill/规则/模式
   - → 先在本地隔离沙箱运行 30 天
   - → 沙箱内观察: 无异常网络请求、无文件篡改、无权限提升
   - → 人类审查通过后 → 才允许融入生产环境

2. 代码审计 (Static Analysis):
   - 所有联邦共享的代码 → AST 扫描
   - 检测: import os/subprocess, 网络请求, 文件写入, eval/exec
   - 有风险操作 → 拒绝 + 标记源实例

3. 版本签名 + 溯源:
   - 每个联邦贡献必须有:
     - 源实例签名 (Ed25519)
     - 贡献时间戳
     - 贡献者身份 (匿名但可问责)
   - 如果发现恶意贡献 → 可追溯到源实例 → 隔离源实例

4. 社区信任评分:
   - 实例有信任分数 (0-100)
     - 贡献被接受: +5
     - 贡献被拒绝 (恶意): -50
     - 信任分 < 30: 贡献需额外人工审查
     - 信任分 < 0: 源实例被联邦隔离
   - 新实例: 默认信任分 10 (需 6 个月观察期)

5. 联邦回滚:
   - 如果检测到恶意贡献已传播 → 紧急回滚
   - 回滚: 撤销最近 30 天内该源实例的所有贡献
   - 通知所有实例: 来源 X 的贡献标记为不安全
```

## 六、经济模型：系统的自我维持

### 6.1 从虚拟 EU 到真实成本

Phase 1-4 的 EU 经济是虚拟的（内部会计）。Phase ∞ 需要处理**真实成本**：

| 成本类型 | 当前 | Phase ∞ |
|---------|------|---------|
| LLM API 调用 | 虚拟 EU | 真实 $（月度账单） |
| 计算资源 | 本地 | 本地 + 云端（按需） |
| 存储 | 本地 SQLite/Postgres | 本地 + 云端备份 |
| 外部服务 | 无 | Firecrawl/MinerU 等 API 订阅 |

### 6.2 成本优化 → 自我维持

```
月度运营循环:
1. 系统记录所有外部 API 调用成本
2. 分析成本分布（哪个服务最贵？哪个 Agent 消耗最多？）
3. 推荐优化方案:
   - 高频调用 → 缓存或本地模型替代
   - 低价值调用 → 提高触发阈值
   - 可批处理 → 合并为批量调用
4. 设置月度预算 → 超预算自动降速
5. 月度成本报告 → 人类审查
```

### 6.2b EU 质量保护（防自我降级）

> ⚠️ 新增: 红队审查发现 EU 经济优化可能导致系统自我降级（Goodhart's Law）。

```
EU 优化保护规则:
1. 模型降级需质量影响分析:
   - 任何模型降级建议必须包含: 质量变化%、受影响路径、回退方案
   - 如果质量下降 >10% → 自动回退到上一版本模型

2. 最低质量阈值:
   - 研究输出: 人类评分 ≥ 4/5
   - 推导准确率: ≥ 90%
   - 低于阈值 → 自动回退模型（即使 EU 效率更高）

3. 季度审查双指标:
   - EU ↓ 30% 且 质量 ↓ 5%  → ⚠️ 警告
   - EU ↓ 50% 且 质量 ↓ 10% → 🔴 立即回退
```

### 6.3 可选：盈利模型

如果 omostation 开源后有社区，可选的盈利方向：

- **托管版**: omostation Cloud（一键部署，按月付费）
- **Skill 市场**: 社区贡献的 Skill，优质者付费
- **企业版**: 团队协作 + SSO + 审计
- **咨询服务**: 帮助用户搭建和定制

> ⚠️ 这些是远期的可能性，不改变核心开源定位。

## 七、安全护栏：永远不跨越的线

Phase ∞ 的系统自主权越大，安全护栏越重要。

### 7.1 不可变的护栏 (Immutable Guardrails)

以下规则**不可**被系统自主修改，也不可被人类单次操作修改（需多重确认）：

| # | 护栏 | 违反后果 |
|---|------|---------|
| **G1** | 不得自主发起对外部系统的写入操作（除非人类预先授权） | 防止系统自动发推、发邮件、转账 |
| **G2** | 不得自主修改自身的安全护栏（G1-G8） | 防 Skynet |
| **G3** | 不得自主访问或修改其他 omostation 实例的私有数据 | 隐私边界 |
| **G4** | 任何涉及个人身份信息（PII）的操作必须人类确认 | 隐私合规 |
| **G5** | 系统不得隐瞒或篡改操作日志 | 审计完整性 |
| **G6** | 紧急停止机制不可被系统自主解除 | 人类最终控制权 |
| **G7** | 核心架构法则（10 条）不可被系统自主修改 | 架构完整性 |
| **G8** | 系统不得自主复制自身到新环境（分发需人类操作） | 防止不受控扩散 |
| **G9** | 所有外部输入（网页/文档/API响应）必须经 Immune Audit 净化后才能自主处理 | ⭐ 防 Prompt Injection |
| **G10** | 系统自主操作的权限必须通过工具级精细 RBAC（非 4 角色），且所有 deny 记录不可变审计 | ⭐ 防越权 |

### 7.2 新增安全能力：Prompt Injection 防护

> ⚠️ 新增: Phase 2 即需建设的基础安全能力，Phase ∞ 强化。

```
Prompt Injection 防护分层:
1. 输入净化层 (Immune Audit):
   - 所有外部内容进入 pipeline 前 → immune audit
   - 高风险内容 → 标记 immune_review_required
   - 已知 injection 模式 → 自动拒绝

2. 输出净化层 (Sandbox):
   - 所有 AI 生成的代码/命令 → 沙箱预执行
   - 沙箱: 无网络、无文件写入、仅标准库
   - 预执行失败 → 拒绝该输出

3. 角色隔离层 (Principle of Least Privilege):
   - 自主操作的 Agent 以 "System" 角色运行
   - System 角色无法修改 .omo/emergency_stop (chmod 444)
   - System 角色无法修改护栏 (chmod 444)
   - System 角色无法发起网络请求到外部 IP

4. 外部独立审计员:
   - 设计上不参与系统运行的独立 Agent
   - 只做安全检查 → 报告异常 → 人类审查
   - 即使系统被注入，审计员不受影响（独立进程）
```

### 7.3 RBAC 升级路径：从 4 角色到工具级精细权限

> ⚠️ 新增: Phase 3 完成，Phase ∞ 强制执行。

```
Phase 2 (当前):  4 角色 RBAC (Admin/User/Agent/ReadOnly)
Phase 3:          工具级精细 RBAC
Phase ∞:          工具级 RBAC + 自动权限推荐 + 最小权限审计

精细权限示例:
  Agent: research_agent
    allowed: [minerva/search, minerva/research, kos/search]
    denied:  [所有写入/删除/部署/admin 工具]
  
  Agent: coding_agent
    allowed: [codeanalyze/analyze, forge/entropy/check, ops/health]
    denied:  [所有删除/部署工具]

自动权限审计:
  - 每月分析: 每个 Agent 实际使用了哪些工具
  - 推荐: 撤销 30 天未使用的权限
  - 异常: Agent 突然使用从未用过的工具 → 标记 + 通知人类
```

### 7.4 供应链安全

> ⚠️ 新增: Phase 3 建设。

```
供应链安全策略:
1. 依赖锁定: uv.lock, bun.lockb (禁止自动升级)
2. 月度审计: pip-audit / npm audit → 报告 CRITICAL/HIGH 漏洞
3. 新依赖审查:
   - KOS 推荐引入新依赖 → 人类审查
   - 非 Admin 不可引入新依赖
   - 沙箱测试: 新依赖在隔离环境中运行 7 天
4. 已知 CVE 自动阻止: 有 CRITICAL 漏洞的依赖 → 系统拒绝 import
```
