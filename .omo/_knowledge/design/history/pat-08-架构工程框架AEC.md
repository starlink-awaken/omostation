## 一、框架总览

### 核心结构

AEC 是原"7步架构方法论 + 5维能力模型"的深度迭代版。核心变化：

| 维度 | 原始框架 | AEC 框架 |
|------|---------|---------|
| **结构** | 线性 7 步 | 6 阶段闭环 + 反馈回路 |
| **需求处理** | 一步跳"理解需求" | 拆为 S1 问题探索 → S2 需求结构化（新增翻译层） |
| **方案设计** | "提出可行的方案" | 强制多方案 → 权衡 → 记录（拒绝单一方案） |
| **质量属性** | "保证系统质量"（太笼统） | QA 场景化：刺激→环境→响应→度量 |
| **扩展性** | 单独一条 | 融入演进机制：技术债兜底 + 适应度函数度量 |
| **反馈** | 无 | S6 → S1 闭环（没有复盘的架构是单程票） |
| **能力模型** | 5 个笼统描述 | 5D × 3 = 15 项可评估、可成长的具体能力 |
| **映射** | 无 | 方法论 × 能力矩阵 + 反模式对照表 |

### 框架示意图

```
                    ┌─────────────────┐
                    │  问题探索 (S1)   │ ◄── 反馈回来重新定义问题
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ 需求结构化 (S2)  │ ◄── 新增：业务→架构的翻译层
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ 抽象建模 (S3)    │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ 方案设计 (S4)    │ ◄── 多方案 + 权衡，拒绝单一方案
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ 增量验证 (S5)    │ ◄── Walking Skeleton，每次验证质量属性
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ 持续演进 (S6)    │ ────► 技术债 + 适应度函数 + 复盘
                    └────────┬────────┘
                             │
                             └──── 反馈闭环 ────► 回到 S1
```

## 二、三条哲学底线

框架不是步骤清单——背后必须有"为什么"。

### 底线 1：架构是风险驱动的，不是步骤驱动的

传统架构方法论"按顺序做"——但现实中你不知道哪些步骤是浪费时间。

**AEC 的回答：风险最高的环节最先做。**

这不是修辞——它改变行为模式：
- 最大风险是"对业务领域理解错位" → 多花时间在 S1/S2，别急着做 S4
- 最大风险是"方案技术上是否可行" → 先做 S5 的 Walking Skeleton，别完善 S3 的抽象模型
- 最大风险是"未来无扩展性" → 先建立 S6 的架构适应度函数

> 启发式：不知道下一步做什么时，问自己——"现在最大的风险是什么？"

### 底线 2：架构是翻译活动，不是创作活动

初级幻觉：架构是"创造"出来的。
高级理解：架构是"翻译"出来的——从利益相关者的意图 → 结构化需求 → 领域模型 → 方案决策。

每次翻译都有信息损失：

```
业务意图（模糊、丰富、充满上下文）
  → 需求文档（精确、结构、丢失语调和权衡）
    → 领域模型（抽象、自洽、失去业务细节）
      → 架构决策（确定、有约束、失去了备选方案）
```

**好的架构不追求"无损翻译"——追求的是"标记翻译中的不确定性"。**

这就是为什么 AEC 要求每阶段有显式交付物——每个交付物是前一次翻译的"出口证据"。

### 底线 3：架构是闭环，不是单程票

最可怕的架构问题不是"选错了技术栈"——而是选错了但没人知道为什么错，也没人纠正。

AEC 的 S6 → S1 反馈闭环意味着：
- 每次复盘都应该质疑当前的假设清单
- 每个架构适应度函数的失效应触发新一轮问题探索
- **架构债务不还，利息会累积**——没有债务追踪系统，你甚至不知道自己欠了多少

## 三、六阶段详解

每个阶段包含：内在矛盾 → 认知模式 → 关键活动 → 交付物 → 深层陷阱

---

### S1：问题探索（Problem Exploration）

#### 内在矛盾：探索深度 vs 时间压力

探索得越深越准，但项目不等人。

#### 认知模式：还原问题空间

不是"收集需求"——需求是被发明的，不是被发现的。在探索前，利益相关者自己也不确定想要什么。

你的任务不是找到一个答案，而是画出一张地图。在这张地图上标出：
- 所有已知的"点"（功能需求、技术约束、合规要求）
- 所有已知的"空白"（不确定性、信息缺口、矛盾意图）
- 所有"可能但不确认"的区域（可选路径、可变约束）

#### 利益相关者地图

标准覆盖 5 类角色：

| 角色 | 他们关心什么 | 他们用什么语言 |
|------|------------|------------|
| **使用者**（日常操作） | 效率、易用、不中断 | "这个按钮在哪" |
| **付费者**（预算决策） | ROI、TCO、风险 | "3年回本" |
| **维护者**（开发/运维） | 可运维、可调试 | "这个代码在哪" |
| **合规者**（法务/安全） | 数据不泄露、可追溯 | "GDPR、等保" |
| **决策者**（CTO/VP） | 战略对齐、市场窗口 | "6个月内上线" |

> 陷阱：90%架构师只在"使用者+维护者"上花90%时间，然后惊讶于方案在决策者那里被毙。

#### 约束的三种类型

| 类型 | 定义 | 示例 | 处理方式 |
|------|------|------|---------|
| **硬约束** | 不可改变 | "必须部署在政务云" | 不可动条件 |
| **软约束** | 可协商但代价已知 | "最好3个月上线" | 标记谈判条件 |
| **隐性约束** | 没说但存在 | "不能和现有系统差异太大" | 需通过场景走查挖掘——最危险 |

#### 交付物

- 上下文图（Context Map）——系统边界、外部依赖、交互协议
- 约束矩阵——硬/软/隐三条清单
- 风险登记册——Top-5 风险及缓解策略

---

### S2：需求结构化（Requirements Structuring）

#### 内在矛盾：精确性 vs 完整性

翻译越精确，丢失的上下文越多。保留越多上下文，结构性越弱。

#### QA 场景化——最核心的工具

把"系统要很快"翻译为可验证的质量属性场景：

```
刺激：用户在08:00高峰点击"报表生成"
环境：系统稳定，CPU<70%
响应：系统在5秒内返回报表首页
度量：P95 < 5秒，成功率 > 99.5%
```

这比"系统要支持高并发"有用一万倍——因为你可以基于这个场景做设计、做测试、做监控。

#### 优先级排序：帮业务方做选择

业务方永远说"都重要"。用稀缺框架帮他们做取舍：

```text
假设我们只能做1件事，你做哪件？
做完这件事，再做第2件，第2件是什么？
做完2件事后，第3件能不能被放弃？
```

关键不是得出"哪个最重要"——而是让决策者看到 trade-off 的存在。

#### 交付物

- 结构化需求模型（用例/用户故事/场景）
- QA 场景清单（每个关键非功能需求对应一个可度量场景）
- 优先级矩阵（Must / Should / Could / Won't）

> 陷阱：架构师跳过 S2 直接进入 S3——觉得自己"已经理解需求"。这是最大的认知偏差——你理解的是你自己对需求的解释，不是利益相关者的意图。

---

### S3：抽象建模（Abstraction & Modeling）

#### 内在矛盾：通用性 vs 具体性

太通用失去指导意义，太具体失去复用价值。

#### 先领域模型，再架构模式

决策序列：

```
业务需求 → 领域模型（通用语言、实体、聚合、限界上下文）
  → 架构风格选择（分层、六边形、CQRS、事件驱动等）
    → 具体技术选型（Spring Boot vs Vert.x）
```

**跳过了领域模型直接选架构模式，就是方案倒推需求。**

检验：你的设计方案里有没有出现业务方听不懂的词？如果有，可能跳过了领域建模。

#### 假设清单——最被忽视的产出

每个架构师都带着大量"默认假设"：
- "数据最终一致是 OK 的"
- "网络延迟在 1ms 以内"
- "团队能维护微服务"

这些假设可能对可能错。关键是你没有意识到你在做假设。

假设清单的价值：
- 明确化、文档化
- 每个假设附带"否定条件"——"如果这个假设不再成立，我们需要做什么？"
- 在 S6 回看假设清单，标记已验证和证伪的

#### 交付物

- 领域模型（实体、值对象、聚合、限界上下文）
- 架构决策标准（什么是"好的"）
- 假设清单

> 陷阱：过度抽象——为模式而模式。"事件溯源+CQRS+六边形架构"——但业务可能只需要一个 CRUD。

---

### S4：方案设计（Solution Design）

#### 内在矛盾：探索广度 vs 决策速度

方案越多选择越好，但决策越慢。

#### 为什么必须多方案？

**只有一个方案的架构不是架构，是信仰。**

> **实战启发：在提出方案之前，先审计现有系统的能力。**
> 在 Agora 网关迭代中，第一步不是写代码，而是读代码——审计了 `registry.py`、`event_bus.py`、`web/app.py`。结论是"80% 已有，只需补 3 个缺口"。这个审计动作避免了至少 2 天的重复工作。
>
> **实践建议**：在 S4 方案设计阶段，花 1-2 小时审计现有代码库中与你的方案相关的模块。你的方案可能不是"新建"，而是"扩展"。

哪怕最后选了 A，在提出 B 和 C 的过程中：
- 发现 A 的潜在弱点（"B 不需要这个中间件，那我们真的需要吗？"）
- 发现隐含假设（"C 依赖最终一致，但产品说必须强一致——确认过了吗？"）
- 让决策可被质疑（"我不选 B 是因为代价是 X，约束是 Y"——而非"感觉 A 好"）

#### Trade-off 分析——不是打分表

打分表幻觉："微服务 85 分，模块化单体 72 分，所以选微服务"——权重从来都是主观的。

真正的 trade-off：让决策者看到每个选项的代价。

| 方案 | 收益 | 代价 | 风险 | 放弃 |
|------|------|------|------|------|
| **微服务** | 独立部署、独立团队 | 分布式事务、网络开销、运维复杂度 | 团队能力不足、基础设施不完善 | 单体简单性 |
| **模块化单体** | 简单、快速启动 | 部署耦合、扩展有限 | 未来拆分的迁移成本 | 独立可扩展性 |
| **Serverless** | 运维最低、弹性最好 | 冷启动、厂商锁定 | 复杂工作流不适用 | 可控性 |

#### ADR——架构师的元工具

一个好 ADR 的结构：

```yaml
标题: "ADR-014：选择 Kafka 作为事件总线"
背景: 为什么要做这个决策
选项: 列出了哪些方案
决策: 选了哪个
理由: 为什么选这个（成本和收益，不是感觉）
后果: 选了之后的影响（好和坏）
后续: 什么情况下需要重新考量
```

> 陷阱：ADR 变成"写一次就归档"的仪式。三个月后没人去翻。每季度复盘应翻一次 ADR。

#### 交付物

- 方案对比表（至少 2-3 个方案）
- ADR 系列
- 风险评估报告

### S5：增量验证（Incremental Validation）

#### 内在矛盾：验证充分性 vs 交付节奏

验证越充分越稳定，但节奏越慢。

#### Walking Skeleton——最小可端到端验证的架构

不是"先搭骨架再填内容"——而是**最小可运行的端到端系统**。

> **实战模式：逐层验证，不等到最后。**
> 在 Agora 网关迭代中，每改一个文件就验证这个文件的 import 和基本功能：
> ```
> 改 agent_card.py  → python3 -c "from agora.agent_card import ..."
> 改 event_bus.py  → python3 -c "publish → 校验 event_id"
> 改 audit_subscriber.py → publish → query → stats → OTel 全线通
> ```
> 这种"逐层验证"比"最后统一跑测试"的问题定位效率高得多——每个错误都精确到刚改的模块。
>
> **实践建议**：在 S5 阶段，每完成一个模块的修改就单独验证这个模块的契约（import + 核心功能）。不要等到所有模块改完才测试。

一个 Walking Skeleton 应覆盖：
- 一条完整的端到端请求链路（API → 业务逻辑 → 数据存储 → 返回）
- 最重要的质量属性验证（安全扫描、性能基线、可部署性）
- CI/CD 管道

**在技术栈完全确定之前，Walking Skeleton 就可以开始做。** S4 和 S5 允许重叠的原因。

#### 质量门禁

| 维度 | 门禁 | 不通过后果 |
|------|------|-----------|
| 功能正确 | 自动化测试 ≥ 80% 通过 | 不允许合并 |
| 性能基线 | P95 不超过阈值 | 标记为性能回归 |
| 安全准入 | 无高危漏洞 | 阻断部署 |
| 可部署性 | CI 流程完整验证 | 标记为部署回归 |

> 陷阱：大爆炸式上线——"做完所有功能再统一测试"。这不是 S5 增量验证，这是 S4 的实现版本。S5 的核心是小步快跑，每次都验证质量属性。

#### 交付物

- 演进路线图（分阶段里程碑）
- 质量门禁清单
- Walking Skeleton

---

### S6：持续演进（Continuous Evolution）

#### 内在矛盾：改进 vs 稳定

每次演进都有破坏风险，但不做改进系统会腐烂。

#### 架构债务四类

不仅仅是代码质量差——架构债是结构性的：

| 债务类型 | 表现 | 度量方式 | 利息 |
|---------|------|---------|------|
| **耦合债** | 改A要改B | 循环依赖检测 | 每次修改成本递增 |
| **技术栈债** | 依赖EOL中间件 | 依赖版本日期、EOL倒计时 | 安全漏洞、无法升级 |
| **文档债** | ADR偏离实际 | ADR vs 代码一致性 | 新人无法理解设计 |
| **自动化债** | CI退化、覆盖率下降 | 构建时间、失败率 | 每次部署焦虑增加 |

#### 架构适应度函数（Fitness Function）

持续演进最重要的工具——自动化检验架构是否退化。

```
适应度函数 = 自动运行的架构检验
```

- **可测试性**：模块间有无契约测试？CI 自动运行
- **可部署性**：commit 到 production 多久？每周追踪
- **耦合度**：依赖图是否有环？每次 PR 检测
- **演进性**：加标准新需求要改多少文件？每月度量

关键是：适应度函数不是你手动检查，是 CI 管道自动执行的。架构退化在 PR 里被 CI 拦住，这个机制才真正起作用。

> 陷阱：从不复盘——或者"复盘的改进从不执行"。最荒诞的模式：每周复盘会，改进项从没进过 sprint backlog。

#### 交付物

- 技术债日志
- 架构适应度报告
- 改进项目清单

## 四、五维能力成长路径

架构师能力不是"有/没有"的二进制——是连续光谱。每个子能力给出 3 级成熟度。

### D1：领域穿透力

| 子能力 | L1（入门） | L2（胜任） | L3（卓越） |
|--------|-----------|-----------|-----------|
| **D1.1 领域建模** | 能理解已有模型，按需修改实体 | 能从 0 设计模型，识别聚合边界 | 能重构混乱模型，设计战略级限界上下文 |
| **D1.2 技术深度** | 在某个方向能独立完成任务 | 能指导该方向的技术选型和诊断 | 能突破已知边界，做出有影响力的贡献 |
| **D1.3 跨域视野** | 了解上下游系统技术方案 | 能快速进入陌生领域并有效贡献 | 能把 A 领域模式应用到 B 领域解决新问题 |

**关键区别**：L2 能做"对的事"；L3 能做"没有先例但对的事"。

### D2：决策力

| 子能力 | L1 | L2 | L3 |
|--------|----|----|----|
| **D2.1 权衡分析** | 能列出优缺点 | 用系统化框架（成本/收益/风险/反向）分析 | 在复杂多层次决策中识别关键驱动因子 |
| **D2.2 不确定性决策** | 充分信息下决策，缺信息就等 | 信息不完备时给出"有条件决策"（附验证计划） | 区分可逆/不可逆，对可逆快速执行 |
| **D2.3 决策记录** | 记录会议纪要 | 用 ADR 系统文档化架构决策 | 建立回溯机制，定期回顾 ADR |

**关键区别**：L2 能做出"理性的决策"；L3 能做"在不确定性下的明智决策"。

### D3：抽象力

| 子能力 | L1 | L2 | L3 |
|--------|----|----|----|
| **D3.1 模式识别** | 识别和使用已知模式 | 从具体问题提取领域模式 | 创造适合上下文的新模式 |
| **D3.2 快速领域切入** | 3 个月熟悉领域 | 1 个月内参与业务讨论并给建议 | 1 周内从文档+代码+访谈了解关键点 |
| **D3.3 系统思维** | 了解模块划分 | 预测局部变化对整体的二阶效应 | 在设计早期预判跨模块影响 |

**关键区别**：L2 在"已知问题"上做得好；L3 在"未知问题"和"跨学科问题"上有效。

### D4：韧性思维

| 子能力 | L1 | L2 | L3 |
|--------|----|----|----|
| **D4.1 接受折中** | 知道没有完美方案，但情感上抗拒 | 主动识别"当前约束下的最优折中点" | 把"折中"作为设计的一阶原则优化 |
| **D4.2 压力下决策** | 压力下回到最熟悉的方案 | 压力下保持"三选法"——快速生3个选项，选代价最小的 | 极端压力下识别 critical path 并做逆转性决策 |
| **D4.3 复盘反刍** | 参与复盘接受反馈 | 主动组织复盘，提取可复用模式 | 建立系统化复盘机制，把模式纳入决策框架 |

**核心转变**：L1 顺境中做得好；L2 逆境不掉链子；L3 把逆境转化为团队能力提升。

### D5：影响力

| 子能力 | L1 | L2 | L3 |
|--------|----|----|----|
| **D5.1 向上对齐** | 能用 PPT 汇报技术方案 | 用决策者语言（成本/风险/ROI）沟通 | 在决策者意识到问题前主动对齐，成其"技术默认顾问" |
| **D5.2 横向推动** | 在团队内推动标准 | 跨团队对齐技术方案，建立共享理解 | 在组织层面建立跨团队技术治理机制 |
| **D5.3 向下赋能** | 解答成员技术问题 | 通过 CR/评审/分享系统性提升团队 | 建立工程文化，让团队自我决策 |

**关键区别**：L2 让别人"听你的"；L3 让别人"不需要你也在做对的事"。

## 五、方法论 × 能力映射

### 映射矩阵

| 阶段 | 高阶能力（必须） | 辅助能力（加分） | 反模式（需警惕） |
|------|-----------------|------------------|-----------------|
| **S1 问题探索** | D1.3 跨域视野, D3.3 系统思维 | D4.2 压力决策 | 跳过利益相关者分析直接写功能列表 |
| **S2 需求结构化** | D1.1 领域建模 | D5.1 向上对齐 | 把需求文档直接转设计，不做翻译 |
| **S3 抽象建模** | D3.1 模式识别, D3.2 快速切入 | D1.2 技术深度 | 过度抽象（为模式而模式）/ 缺乏抽象（全是 if-else） |
| **S4 方案设计** | **D2.1 权衡分析**, D2.2 不确定性决策 | D5.2 横向推动 | 只有一种方案（没有比较就没有架构） |
| **S5 增量验证** | D4.1 接受折中 | D5.3 向下赋能 | 大爆炸式上线 / 只做功能验证不做架构验证 |
| **S6 持续演进** | **D4.3 复盘反刍** | D2.3 决策记录 | 从不复盘 / 复盘的改进从不执行 |

### 深层规律

```
前三个阶段（S1/S2/S3）——需要"输入处理"能力
  → D1 领域穿透力（理解输入）
  → D3 抽象力（处理输入）

后三个阶段（S4/S5/S6）——需要"输出交付"能力
  → D2 决策力（做出判断）
  → D4 韧性思维（承受后果）
  → D5 影响力（让他人接受）

贯穿始终：D3 抽象力和 D5 影响力——从 S1 到 S6 都需要。
```

### S1 为什么必须 D1.3 + D3.3

问题探索最需要"宽度"——看到系统边界之外。跨域视野不够（D1.3），你会把探索范围定太窄，错过关键约束。系统思维不够（D3.3），你不会意识到一个约束对系统其他部分的影响。

**常见失败模式**：技术深度很高但跨域很窄的架构师，在 S1 会犯"把技术方案当成需求"的错误——"这个系统需要一个消息队列"（需求还没分析完）。

### S4 为什么 D2.1 是排他性的"必须"

S4 是整个 AEC 中唯一要求**两个高阶能力**的阶段。原因是：

方案设计是架构师最孤独的工作阶段。你可能是唯一做过这个方案的人，需要在没有人告诉你对错的情况下判断，需要在不充分的信息中做出有条件的决策。

**如果你没有权衡分析能力，你的方案是基于直觉的、不可传递的、不可被挑战的。**

## 七、使用水平三阶段

### 水平 1：照着做（能跑就行）

- 记住 6 个阶段的名称
- 每个阶段知道要产出一个什么交付物
- 遇到问题能回头查 S1 是否漏了关键约束
- **效果**：从"凭感觉做架构"到"有条理地做架构"

### 水平 2：能调整（理解为什么）

- 理解三条哲学底线
- 知道什么时候可以跳过一个阶段（因为风险分析表明不需要）
- 能根据自己的上下文调整每个交付物的详细程度
- **效果**：从"跟着框架走"到"框架跟着我的问题走"

### 水平 3：能再造（框架内化了）

- 不用查文档也知道下一步该做什么
- 能在团队中教别人用这个框架
- 能基于真实项目经验给框架的每个阶段补充反模式和启发式
- **效果**：框架已经消失在你每个决策的背后。你不再提"AEC 说"——因为 AEC 已经是你的默认思维模式

## 八、框架自反性

一个不能用在自身上的框架，不是元框架。

### AEC 应用于 AEC 本身

| AEC 阶段 | 对框架本身的思考 |
|---------|----------------|
| **S1 问题探索** | 这个框架解决什么问题？利益相关者是个人架构师，不是企业架构部门。约束：不能太复杂（个人没空学100页方法论） |
| **S2 需求结构化** | 框架的"QA场景"：一个架构师在 2 小时内学会并能在项目中开始使用。度量：用了之后比以前少犯哪些错误 |
| **S3 抽象建模** | 框架的领域模型：AEC 循环、5D 能力、15 子能力、3 级成熟度——核心实体 |
| **S4 方案设计** | 框架可有多"方案风格"：简约版（6阶段+5D名称）vs 完整版（含成熟度+映射+自反性） |
| **S5 增量验证** | 框架的 Walking Skeleton——本文档。需在真实场景验证 |
| **S6 持续演进** | 框架本身的 ADR——何时、为什么、做了什么改动 |

> **实战案例：ADR-016 Agora Agent Card 标准选型**
>
> ```
> 背景: 需要让 Agora 上注册的服务可以被外部 Agent 通过标准协议发现
> 选项:
>   A. 自创元数据格式
>   B. 采用 A2A Agent Card 规范（Linux 基金会标准）
>   C. 采用 MCP 原生发现机制
> 决策: B
> 理由: A2A 是 Linux 基金会标准，Agent 互通将是核心需求。
>       自创格式需要外部 Agent 适配，A2A 有现成的 /.well-known 发现机制
> 后果:
>   - 正向：外部 A2A 兼容 Agent 可直接发现 Agora 服务
>   - 反向：需要将内部 ServiceConfig 映射为 Agent Card 格式
> 后续: A2A 规范更新时同步更新 agent_card.py 的 to_dict()
> ```

## 九、和现有方法论的关系

AEC 不取代现有方法论——它回答了一个现有方法论都不回答的问题：**"一个架构师从接到任务到系统退役，应该怎么思考和行动？"**

| 方法论 | 定位 | 和 AEC 的关系 |
|--------|------|-------------|
| **TOGAF** | 企业级架构管理（组织、流程、治理） | AEC 是 TOGAF 中"架构师个体工作方法"的落地，但不是 TOGAF 本身 |
| **DDD** | 领域建模方法 | AEC 的 S3（抽象建模）可直接用 DDD。AEC 回答了"在什么时候用 DDD" |
| **C4 Model** | 可视化方法 | AEC 的 S2/S3 可用 C4 画图。AEC 回答了"画什么样的图、什么时候画" |
| **敏捷/Scrum** | 项目管理方法 | AEC 回答了"做什么"，敏捷回答了"怎么做"。互补 |
| **UML** | 建模语言 | AEC 不指定用 UML 还是 C4。你怎么表达模型都可以 |

## 十、附录：与原始框架的差异对照

| 维度 | 原始框架 | AEC 迭代后 |
|------|---------|-----------|
| **结构** | 线性 7 步 | 6 阶段闭环 + 反馈回路 |
| **需求处理** | 一步跳"理解需求" | 拆为 S1 问题探索 → S2 需求结构化（新增翻译层） |
| **方案设计** | "提出可行的方案" | 强制多方案 → 权衡 → 记录（拒绝单一方案） |
| **质量属性** | "保证系统质量"（太笼统） | QA 场景化：刺激→环境→响应→度量 |
| **扩展性** | 单独一条 | 融入演进机制：技术债兜底 + 适应度函数度量 |
| **反馈** | 无 | S6 → S1 闭环（没有复盘的架构是单程票） |
| **能力模型** | 5 个笼统描述 | 5D × 3 = 15 项可评估、可成长的具体能力 |
| **映射** | 无 | 方法论 × 能力矩阵 + 反模式对照表 |
| **成长路径** | 无 | 每个子能力 3 级成熟度（L1→L2→L3） |
| **使用水平** | 无 | 照着做 → 能调整 → 能再造 |
| **自反性** | 无 | 框架应用于框架本身 |
