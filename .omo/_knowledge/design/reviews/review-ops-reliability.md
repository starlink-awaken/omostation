# SRE 运维可靠性审阅报告 — Task Center

> **审阅对象**: `task-center-requirements.md` v0.1 (草案)
> **审阅日期**: 2026-05-31
> **审阅角色**: 运维可靠性工程师 (SRE)
> **审阅范围**: 架构设计、数据持久性、监控告警、容错降级、资源管理、变更管理

---

## 摘要

Task Center 需求文档在业务逻辑和功能设计上充分吸收了历史事故教训（断裂积累、无统一观测），四层架构（Registry → Scheduler → Executor → Observability）分层清晰。但从运维可靠性角度，**文档存在 3 项 CRITICAL 级别的设计缺口**：cron-service 单进程 SPOF 缺少热备方案、运行记录缺少原子写入保障、tick 循环缺少 backpressure 机制。此外，监控告警体系仅有波形架构（Wave 3 才落地告警），数据持久性和变更管理的容错路径不够完善。**系统可靠性成熟度评估为 v0.5（MVP 级别），需要在前 3 个里程碑内补齐核心可靠性短板，否则生产运行风险偏高。**

---

## CRITICAL 发现

### C1: cron-service 单进程 SPOF — 无热备/冷备切换

| 项目 | 内容 |
|------|------|
| **故障场景** | cron-service 进程崩溃（OOM、segfault、Python 运行时异常），全部调度任务停止 |
| **影响组件** | Scheduler 层 (tick loop)、Executor 层 (runner pool)、MCP 工具集 |
| **严重级别** | CRITICAL |
| **文档状态** | 风险矩阵 R1 已识别，缓解措施为 launchd 自动重启 |
| **问题分析** | (1) launchd 重启间隔实测 1-3s，但重启期间所有到期任务全部丢失 —— 无任务持久化队列，到期判断基于内存中的 tick 轮次。假设 15s tick，崩溃发生在第 14s，重启后距离下一个 tick 最多丢失 29s 的任务窗口。(2) 无健康检查探针区分"进程活着"和"调度正常"——tick loop 卡死（如 SQLite 锁等待无限期阻塞）时，进程仍在运行但调度已停。launchd 不会重启。 (3) chronyd/ntpd 回拨时钟可能导致 tick 间隔异常放大或跳过 |
| **建议修复** | **短期（MVP 必做）**: 增加 tick loop 健康探针——每个 tick 写入 `_delivery/task-center/heartbeat.json`，观测层独立监控该文件的更新时间戳，滞后超过 2 个 tick 间隔（30s+）则触发告警。**中期**: 实现进程内 watchdog，tick loop 用 `signal.SIGALRM` 或 `threading.Timer` 自监视，超时未完成主动 crash（fail-fast），利用 launchd 重启。**长期（联邦阶段）**: 引入备用调度进程或事件驱动替代纯轮询。 |

### C2: 运行记录写入缺乏原子性保障

| 项目 | 内容 |
|------|------|
| **故障场景** | 任务执行完成写入 `_delivery/task-center/runs/{task_id}/{timestamp}.json` 时进程崩溃/磁盘满，留下不完整 JSON 文件 |
| **影响组件** | 观测层（运行记录）、后续依赖运行记录的审计/分析工具 |
| **严重级别** | CRITICAL |
| **文档状态** | 4.5.1 节定义了写入格式，无原子写入策略描述 |
| **问题分析** | (1) `json.dump` → `open().write()` 非原子操作。Python 默认 write buffer 可能导致部分写入。 (2) 半截 JSON 文件无法被下游解析，导致运行记录链断裂。 (3) 恢复时无法区分"写入中崩溃"和"任务确实未执行"。 |
| **建议修复** | **强制使用原子写入模式**: 写入临时文件 `{timestamp}.json.tmp` → `os.fsync()` → `os.rename()` 到 `{timestamp}.json`。启动时扫描并清理残留 `.tmp` 文件。**增加写入隔离**: 将运行记录写入抽象为 `RunRecorder` 类，统一原子写入 + 重试 + 完整性校验。 |

### C3: Tick 循环缺失 Backpressure 机制

| 项目 | 内容 |
|------|------|
| **故障场景** | 某 tick 中 50 个 cron 任务同时到期，执行耗时 60s（远超 TICK_INTERVAL=15s），下一个 tick 的到期任务叠加到当前未完成的任务队列上 |
| **影响组件** | Scheduler 层、Executor 层（runner pool） |
| **严重级别** | CRITICAL |
| **文档状态** | 7.3 验收项提到"连续 tick 不累积延迟"但无具体实现机制 |
| **问题分析** | (1) 文档未说明 tick loop 的实现模型：是固定间隔 `while True: sleep(15)` 还是补偿式 `next_tick = max(now + 15, last_tick_end + 15)`？前者必然导致 tick 漂移累积。 (2) Semaphore（并发数 4）只控制并行度，不控制队列深度。当 50 个任务排队时，内存中会积累大量 `asyncio.Task` 或 `threading.Thread` 对象。 (3) 单任务雪崩保护（1 分钟最多 4 次触发）是任务级限制，但非系统级 backpressure——系统层面没有"总排队超过 N 个则跳过低优先级 tick"机制。 |
| **建议修复** | **(1) 补偿式 tick**: 使用 `next_tick_time = start + interval` 而非 `start + interval`（固定间隔）。如果本次 tick 耗时 60s，自动跳过中间 tick，下一 tick 从 `start + 60` 开始。 **(2) 队列上限**: Semaphore 之外增加**待执行队列长度上限**（建议 100），超过上限时：`log_warning` → 跳过非关键任务 → 触发告警。 **(3) 验收测试必须包含**: 构造 50 个同时到期任务，验证 tick 不累积、无 OOM。 |

## HIGH 发现

### H1: 监控告警体系完整性不足

| 项目 | 内容 |
|------|------|
| **故障场景** | iLink 微信服务宕机或网络不可达，告警完全丢失；SRE 需排查历史故障但无历史指标数据 |
| **影响组件** | 观测层（告警通道） |
| **严重级别** | HIGH |
| **文档状态** | 4.5.3 告警仅提到 iLink/邮件通知，无通道降级策略 |
| **问题分析** | (1) **单告警通道依赖**: 仅依赖 iLink 微信。iLink 服务异常或网络中断时告警静默。 (2) **无告警历史**: 告警事件本身无持久化存储。 (3) **无 SLO/SLI 仪表板**: 文档 2.3 定义了 99.9% 调度精度目标，但无 SLI 采集方法、无 SLO 燃烧速率告警。 (4) **无告警静默/抑制**: 连续失败触发告警风暴时无降噪机制。 |
| **建议修复** | **(1) 多通道**: iLink 作为主通道 + 本地通知（terminal-notifier/macos-notification）作为保底通道。 **(2) 告警持久化**: 告警事件写入 `_delivery/task-center/alerts/`（原子写入）。 **(3) SLI 采集**: 定义 `schedule_precision_sli = 实际触发时间 - 计划触发时间`，每分钟输出一个指标点。99.9% SLO = 99.9% 的 tick 延迟在 15s 以内。 **(4) 告警抑制**: 同一任务连续失败仅首次 + 每 1h 重复通知，避免告警轰炸。 |

### H2: registry.yaml 变更管理的恢复路径模糊

| 项目 | 内容 |
|------|------|
| **故障场景** | 人为误操作删除/错误修改 registry.yaml 中的关键任务定义 |
| **影响组件** | 注册层、调度层 |
| **严重级别** | HIGH |
| **文档状态** | 6.1 R8 提到 Git 冲突处理，6.2 提到回滚 registry.yaml |
| **问题分析** | (1) Git 回滚期间调度引擎需要继续工作——回滚需要重新加载 registry.yaml 并重建 SQLite cache，该过程未保证零中断。 (2) 无变更审批流——谁有权修改 registry.yaml？文档提到"自治执行，失败告警"，但注册层的变更（创建/删除/暂停任务）应至少记录审计日志。 (3) 错误修改的回滚依赖 Git 操作，但如果修改后未及时 commit，Git 无法回滚。 |
| **建议修复** | **(1) SVT（Schema Validation Test）**: 每次 registry.yaml 修改后，MCP 层执行前自动 schema 校验 + dry-run，校验不通过禁止写 cache。 **(2) 变更审计日志**: MCP 的 `task_create/update/delete` 每次操作写入 `_delivery/task-center/audit/`（谁、什么时间、改了哪些字段、diff）。 **(3) 快照备份**: `cron.db`（SQLite cache）在每次 sync 前自动备份到 `cron.db.bak`，回滚时用 cache 备份 + registry.yaml 历史版本双保险。 |

### H3: 运行记录无限增长 — 无清理/归档策略

| 项目 | 内容 |
|------|------|
| **故障场景** | Task Center 运行数月后，`_delivery/task-center/runs/` 目录膨胀至数万文件，磁盘满/inode 耗尽 |
| **影响组件** | 交付面（_delivery/）、磁盘 |
| **严重级别** | HIGH |
| **文档状态** | 4.5.1 定义写入格式和路径，无清理策略 |
| **问题分析** | (1) 默认超时 120s，每个 tick 最多可产生 4 个运行记录（并发数）。若 100 个任务每小时触发一次，每天 2400 个运行记录，一年约 87 万条记录。 (2) 文件数量过多影响 `ls` 和文件系统性能。 (3) 无归档机制——历史运行记录可能对审计有价值，但不能无限保留在线。 |
| **建议修复** | **(1) 分层保留策略**: 在线保留最近 7 天运行记录、近 30 天摘要（仅 status/duration/exit_code）、30 天以上归档到压缩文件（`.tar.gz`）。 **(2) 配额告警**: 当 `_delivery/task-center/runs/` 文件数超过 10,000 或磁盘使用率超过 80% 时触发告警。 **(3) 自动清理定时任务**: 内置一个 `task-center-housekeeping` cron job（注册在 registry.yaml 自身），每日清理过期记录。 |

## MEDIUM 发现

### M1: 事件监听（fs event）资源管控不完整

| 项目 | 内容 |
|------|------|
| **故障场景** | event 类型任务监听大型目录（如 `~/Workspace/`），kqueue/FSEvents watch 数量超限或 CPU 飙升 |
| **影响组件** | Scheduler 层（event listener） |
| **严重级别** | MEDIUM |
| **文档状态** | 4.1.5 定义了 watch 目录、debounce、glob 模式，6.1 R3 提到资源消耗，回退方方案为 interval 轮询 |
| **问题分析** | (1) macOS kqueue 的 watch 数默认较低（`sysctl kern.maxkernlevents` 典型值 4096），每个 watched 子目录消耗一个 fd。递归监听深度未定义。 (2) FSEvents 对大型目录（10 万+文件）的首次扫描消耗大量 CPU。 (3) 回退到 interval 轮询是好的，但切换逻辑未定义——是检测到错误后自动回退，还是手动切换？ |
| **建议修复** | **(1) 监听限制**: 限制 `max_watches_per_event`（建议 1000），超过时告警并拒绝注册。 **(2) 自动降级**: 检测到 kqueue fd 耗尽或 FSEvents 错误时自动回退到 interval 轮询（polling interval 可配置，默认 60s）。 **(3) 首次扫描优化**: 在 debounce 窗口内完成扫描，避免一次性 CPU 峰值。 |

### M2: 并发控制队列行为不透明

| 项目 | 内容 |
|------|------|
| **故障场景** | 整点时刻 20+ cron 任务同时到期，Semaphore(4) 限制下 16 个任务在队列中等待，等待时间不可预测 |
| **影响组件** | Executor 层（runner pool） |
| **严重级别** | MEDIUM |
| **文档状态** | 4.5.3 定义了单任务雪崩保护，4.2.2 定义了任务级超时，6.1 R6 承认了并发风险 |
| **问题分析** | (1) Semaphore(4) 默认值是合理的，但排队策略未定——是 FIFO 还是按优先级？关键任务（如知识索引）不应与低优任务（如日志清理）争用同一个 semaphore。 (2) 文档未说明 Semaphore 是跨所有任务类型共享，还是按类型分组（如 cron/event/webhook 分开）。event 任务延迟敏感但 cron 任务量大，混用会导致 event 处理延迟偏高。 (3) 无"最大排队等待时间"告警——队列中任务等待超过 5 分钟应触发告警。 |
| **建议修复** | **(1) 优先级队列**: 引入 `priority` 字段（默认 5，1-10 范围），Semaphore 按优先级出队，高优先级任务插队。 **(2) 分类 Semaphore**: cron 和 longrun 共享 4，event 独立 2，webhook 独立 2——防止一种类型堵塞其他类型。 **(3) 等待时间告警**: 任务在队列中等待超过 `timeout * 2` 或 300s（取较小值）时触发告警。 |

### M3: SQLite 损坏恢复期间服务不可用

| 项目 | 内容 |
|------|------|
| **故障场景** | cron.db SQLite 文件损坏，sync 工具尝试从 registry.yaml 重建，重建期间调度不可用 |
| **影响组件** | 注册层（SQLite cache） |
| **严重级别** | MEDIUM |
| **文档状态** | 文档提到 SQLite 是 cache，损坏后从 registry.yaml 重建 |
| **问题分析** | (1) 重建需要锁定 registry.yaml 并逐条注册，500 个任务重建耗时预估可能超过 2s（验收目标是 < 2s），此期间调度 pause。 (2) 如果重建过程中 registry.yaml 本身被意外修改（Git pull），可能进入不一致状态。 (3) SQLite WAL 模式未启用——默认 journal 模式在并发读写下更易损坏。 |
| **建议修复** | **(1) WAL 模式**: SQLite 启用 `PRAGMA journal_mode=WAL` + `PRAGMA synchronous=NORMAL`，提高并发性和崩溃安全性。 **(2) 增量重建**: 损坏时先使用 `PRAGMA integrity_check` 确认损坏范围，只重建坏掉的表。 **(3) 双 cache**: 保留两份 cache（`cron.db` 和 `cron-fallback.db`），primary 损坏时自动切换 fallback，后台重建 primary。 |

### M4: webhook 速率和超时配置脱节

| 项目 | 内容 |
|------|------|
| **故障场景** | webhook 请求处理超时 10s，但单个流式响应可能超过 10s；同 path 速率限制 10 次/分钟 → 平均 6s/次，若请求集中到达（如 CI 批量触发），第 11 次请求被直接丢弃但无反馈 |
| **影响组件** | Scheduler 层（webhook listener） |
| **严重级别** | MEDIUM |
| **文档状态** | 4.1.4 定义了 webhook 超时 10s、速率限制 10 次/分钟 |
| **问题分析** | (1) 速率限制被拒绝的请求应返回 `429 Too Many Requests` + `Retry-After` header，而非静默丢弃。 (2) 10s 超时对于需要执行脚本的 webhook 偏短——注册回调耗时短但执行脚本可能超过 10s（尤其是 deployment 类）。建议区分注册超时和执行超时。 (3) 无 burst 容量——允许短时爆发然后平滑。 |
| **建议修复** | **(1) 规范速率限制响应**: 被限流的请求返回 HTTP 429 + `Retry-After: 6`（秒），body 中说明限流原因。 **(2) 区分超时层级**: webhook listener 自身超时 5s（注册回调），执行超时沿用任务级 timeout 配置（默认 120s）。 **(3) 令牌桶**: 使用令牌桶算法替代固定窗口，允许短时 burst（如 5 次突发），长期平均 10 次/分钟。 |

## LOW / 建议

### L1: 磁盘满场景无预案

| 项目 | 内容 |
|------|------|
| **故障场景** | 磁盘使用率达到 100%，任何文件写入（运行记录、registry.yaml sync、log）均失败 |
| **影响组件** | 全部（跨层影响） |
| **严重级别** | LOW |
| **建议** | (1) 守护进程启动时检查磁盘可用空间，低于 `min_disk_gb: 1`（默认 1GB）时进入安全模式：只执行存量任务，不写入运行记录（缓冲到内存，空间恢复后写入），不接收新 webhook。 (2) 每天第一次 tick 执行磁盘空间检查，写入健康指标。 |

### L2: 时钟回拨/偏移保护

| 项目 | 内容 |
|------|------|
| **故障场景** | NTP 校正导致系统时钟回拨数秒（常见），cron 调度可能跳过或重复触发 |
| **影响组件** | Scheduler 层 |
| **严重级别** | LOW |
| **建议** | (1) 使用单调时钟（`time.monotonic()`）计算 tick 间隔，避免受系统时间调整影响。 (2) cron 表达式解析时记录 `last_fire_time`，如果系统时间跳变超过 `tick_interval * 2`，跳过该 tick 并告警（而非追赶调度）。 |

### L3: 网络中断（delivery 到 iLink）降级行为

| 项目 | 内容 |
|------|------|
| **故障场景** | Task Center 到 iLink 服务网络中断，deliver: origin 的任务结果无法投递 |
| **影响组件** | 执行层（投递分发） |
| **严重级别** | LOW |
| **建议** | (1) 投递失败时自动降级为 `deliver: local`，本地保存结果，网络恢复后重试投递（最多 3 次）。 (2) 投递队列持久化到本地 SQLite —— 避免进程重启丢失未投递结果。 |

## 结论

### 系统可靠性成熟度评估

| 维度 | 评分 (1-5) | 说明 |
|------|------------|------|
| 架构弹性 | 2/5 | 单进程 SPOF 未解决，tick loop 无 backpressure |
| 数据持久性 | 3/5 | SSOT 设计好，但写入非原子 + 无归档策略 |
| 监控可观测 | 2/5 | 告警依赖单通道、无 SLO/SLI、无仪表板 |
| 容错降级 | 2/5 | 事件监听有回退方案，但全局降级策略缺失 |
| 变更管理 | 3/5 | Git 驱动了版本管理，但无审计日志和审批流 |
| 资源管理 | 2/5 | 并发控制基础有了，但排队和优先级缺失 |
| **综合** | **2.3/5** | **MVP 可接受，生产运行需先补齐 P0/P1 改进项** |

### 关键结论

1. **MVP 可以推进**，但必须在前三个里程碑（MVP Launch · Wave 2 启动 · Wave 3 告警）之前解决 C1/C2/C3 三个 CRITICAL 级问题。
2. **文档当前最缺乏的是定量设计**——99.9% 可靠性目标很好，但没有 SLI 定义、没有故障恢复时间（RTO）目标、没有数据丢失容忍（RPO）目标。建议在 2.3 节补充：
   - RTO < 30s（cron-service 崩溃恢复）
   - RPO < 1 tick 间隔（最多丢失 15s 的调度事件）
   - SLI 采集方式（每条运行记录自动产出调度延迟指标）
3. **告警体系是最大设计缺口**——Wave 3（2026-06~08）才落地告警意味着 MVP 期间无告警。建议在 Wave 2 MVP 中至少包含基础告警（连续失败通知 + 健康探针），而非推迟到 Wave 3。
4. **混沌工程应纳入验收标准**——建议在 7.5 迁移验收之后增加"7.6 可靠性验收"，包含 kill -9 测试、SQLite 损坏恢复测试、磁盘满行为测试。
